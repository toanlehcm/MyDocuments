Workflow Thực Tế: Kiến Trúc Hexagonal

Quên đi các "hình lục giác" phức tạp. Hãy xem chính xác một Request API đi qua các Layer nào cho đến khi chạm vào Database.

❓ Giải đáp nhanh các câu hỏi cốt lõi:

1. Server sẽ tạo bao nhiêu layer?

Về cơ bản là 3 vùng chính:
1. Layer hứ́ng Request (Controller)
2. Layer Core (Xử lý nghiệp vụ)
3. Layer Giao tiếp DB (Repository).

2. Layer nào tiếp xúc với DB (mysql, mongodb...)?

Layer thứ 3: Secondary Adapter (Repository). Đây là nơi duy nhất chứa code thực thi Database như foodModel.findById(id).

Nhấp vào từng bước dưới đây để xem Code và Chức năng chi tiết

Nơi xử lý Payload và các quy tắc kinh doanh.

❓ Layer này làm gì? Nhận payload từ Layer 1 truyền sang. Nó không quan tâm đến Web hay HTTP. Nó chỉ tập trung tính toán: Đơn hàng này có hợp lệ không? Tổng tiền là bao nhiêu?
❓ Chuyện gì xảy ra tiếp theo? Sau khi xử lý xong, nó không tự lưu vào DB. Nó gọi một "Interface" (Bản hợp đồng) để yêu cầu: "Tôi cần lưu dữ liệu này".
OrderUseCase.js JavaScript
class OrderUseCase { // Inject Repository (Interface) vào constructor(orderRepository) { this.repo = orderRepository; } execute(payload) { // 1. Kiểm tra nghiệp vụ if (!payload.foodId || payload.qty <= 0) { throw new Error("Dữ liệu đơn hàng không hợp lệ!"); } // 2. Tính toán const totalAmount = payload.qty * 50000; const finalOrder = { ...payload, totalAmount }; // 3. YÊU CẦU lưu DB thông qua Interface (không dùng mongoose ở đây) this.repo.save(finalOrder); return finalOrder; } }
