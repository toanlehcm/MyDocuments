Kiến Trúc Ports & Adapters

(Kiến trúc Hexagonal / Lục giác)

Giải pháp bảo vệ Logic nghiệp vụ cốt lõi bằng cách giao tiếp với các công cụ bên ngoài (UI, Database, API) thông qua các Cổng (Ports) và Bộ chuyển đổi (Adapters). Framework hay Database chỉ là "plug-in" có thể tháo lắp dễ dàng.

1. Định nghĩa Port (Trong Lõi)

Lõi chỉ định nghĩa ra một "Ổ cắm" (Interface) yêu cầu phải có hàm lưu dữ liệu, không quan tâm lưu ở đâu.

interface IOrderRepository {
  save(order: Order): void;
}
🔌
🔗

Adapter Cũ (MySQL)

implements IOrderRepository

🐘
CHUYỂN ĐỔI DỄ DÀNG ⬇️

Adapter Mới (MongoDB)

implements IOrderRepository

🍃

Kết quả: Bạn chỉ cần tạo ra file MongoOrderRepository mới chuẩn hóa theo IOrderRepository và "cắm" nó vào hệ thống lúc khởi chạy. Lõi ứng dụng (Application Core) không thay đổi dù chỉ 1 dòng code.

Hexagonal (Ports & Adapters) vs Layered MVC truyền thống

🎯 NÊN ÁP DỤNG KHI:

  • ✔️ Dự án dài hạn, thay đổi liên tục: Hệ thống cần sống sót qua nhiều thế hệ công nghệ (đổi framework frontend, đổi DB, tích hợp API mới liên tục).
  • ✔️ Domain Logic phức tạp: Quy tắc tính toán, nghiệp vụ cốt lõi quá quan trọng và rắc rối, cần được cô lập tuyệt đối để dễ hiểu và không bị tác động bởi UI.
  • ✔️ Yêu cầu Unit Test nghiêm ngặt: Giúp Test logic cốt lõi nhanh chóng mà không cần mock các thư viện DB hay chạy server.

⚠️ CÂN NHẮC KHI:

Dự án quá nhỏ (MVP, CRUD cơ bản) hoặc có Deadline cực ngắn. Kiến trúc này yêu cầu tạo nhiều file (Interfaces, DTOs, Mappers), gây ra tính "cồng kềnh" (boilerplate) ở giai đoạn đầu.
