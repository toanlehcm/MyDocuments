# PTE-9503: Fix Image Upload Orientation — Implementation Plan v2

## #1. Nhận xét góp ý của Dev A

Dev A nói:
> *"Portrait photos taken on phones often appear rotated in web browsers. That's because phones save images in a default horizontal (landscape) format and rely on hidden metadata—called EXIF tags—to tell the device to display them vertically. While photo gallery apps usually read this metadata correctly, many browsers and websites ignore it (`<img>` tag), which causes the pictures to appear in their original sideways orientation."*

### Nhận xét:

| Phần | Đánh giá | Giải thích |
|------|----------|------------|
| "Phones save in landscape format + EXIF tag" | ✅ Đúng | Camera sensor vật lý luôn nằm ngang, dùng EXIF orientation tag để chỉ hướng xoay |
| "Many browsers ignore EXIF in `<img>` tag" | ⚠️ **Đúng trước 2020, sai từ 2020** | Từ Chrome 81 (April 2020), tất cả Chromium-based browsers **tự động** đọc EXIF và xoay ảnh. Firefox, Safari cũng đã hỗ trợ. |
| CSS `image-orientation: from-image` | ⚠️ **Không cần nữa** | Đây đã là default behavior của mọi browser hiện đại. Thêm CSS này không giải quyết vấn đề canvas. |
| JS đọc EXIF + Canvas xoay | ✅ **Đây chính là code hiện tại** | `imageFactory.js` đang dùng cách này — và chính nó gây ra bug double rotation |
| Backend hard-rotate | ❌ **Không nên** | Thêm dependency server-side (sharp/jimp), tăng latency upload, phải sửa Server code |

### Kết luận về góp ý của Dev A:

Dev A **đúng về nguyên lý cơ bản** (phone lưu landscape + EXIF), nhưng **thiếu context quan trọng**: từ Chrome 81+ (2020), browser đã TỰ ĐỘNG xoay ảnh theo EXIF. Code `resetOrientation` của PTE được viết cho thời browser CHƯA hỗ trợ auto-rotate. Giờ browser đã hỗ trợ → code xoay THÊM lần nữa → **double rotation**.

Ba giải pháp Dev A đề xuất:
1. **CSS `image-orientation`** → Không giải quyết được (vấn đề ở canvas, không phải `<img>`)
2. **JS + Canvas xoay** → Chính là code hiện tại, và chính nó gây bug
3. **Backend hard-rotate** → Overkill, thêm complexity không cần thiết

## #2. Khi nào một hình có EXIF?

EXIF (Exchangeable Image File Format) được **tạo tự động bởi thiết bị chụp** tại thời điểm chụp ảnh.

| Nguồn ảnh | Có EXIF? | Chi tiết |
|-----------|----------|----------|
| Camera điện thoại (Samsung, iPhone...) | ✅ Luôn có | Chứa: orientation, model, date, GPS, exposure... |
| Camera DSLR/Mirrorless | ✅ Luôn có | EXIF đầy đủ nhất |
| Screenshot | ❌ Không | PNG, không có camera metadata |
| Ảnh tạo bởi design tool (Photoshop, Figma) | ❌/⚠️ Hiếm | Thường export với orientation=1 hoặc không có EXIF |

**Chỉ hỗ trợ cho JPEG và TIFF. PNG KHÔNG có EXIF.**

📎 Links:
- EXIF Spec chính thức (CIPA): https://www.cipa.jp/std/documents/download_e.html?DC-008-Translation-2023-E
- ExifTool Tag Reference: https://exiftool.org/TagNames/EXIF.html
- Wikipedia EXIF: https://en.wikipedia.org/wiki/Exif
- MDN image-orientation: https://developer.mozilla.org/en-US/docs/Web/CSS/image-orientation

## #3. EXIF có trong hình download từ web không?

**Tùy nguồn.** Hầu hết web/social media **xóa EXIF** khi upload/download:

| Nguồn download | Giữ EXIF? | Orientation thực tế |
|----------------|-----------|---------------------|
| Facebook, Instagram, WhatsApp | ❌ Strip EXIF | orientation = 1 hoặc không có |
| Twitter/X | ❌ Strip EXIF | orientation = 1 |
| Google Images (search) | ⚠️ Tùy nguồn gốc | Thường orientation = 1 (đã bake pixel đúng chiều) |
| Flickr | ✅ Giữ EXIF | Có thể có orientation ≠ 1 |
| Download trực tiếp từ photographer | ✅ Giữ EXIF | Có thể có orientation ≠ 1 |

**Bằng chứng từ debug thực tế của anh:**
- Hình chụp Samsung: `orientation = 6` → CÓ EXIF rotation
- Hình download Google: `orientation = 1` → KHÔNG cần rotation

📎 Links:
- Adobe EXIF stripping: https://helpx.adobe.com/photoshop/using/file-info.html
- Stack Overflow - social media strip EXIF: https://stackoverflow.com/questions/11918596/does-facebook-strip-exif-data

## #4. Solution đề xuất — Không cần hardcode test image

### Ý tưởng của anh rất đúng hướng. Em đề xuất **3 options** từ đơn giản → phức tạp:

---

### Solution A: Dimension-based detection (✅ RECOMMENDED)

**Ý tưởng:** Thay vì hardcode test image, so sánh EXIF orientation với `img.width/height` thực tế sau khi load. Nếu kết quả cho thấy browser đã tự xoay → skip transform.

**Logic:**
- EXIF orientation 5-8 = xoay 90° hoặc 270° = raw pixel data là **landscape** (width > height)
- Nếu sau khi load, `img.height > img.width` (đã portrait) → browser đã tự xoay → SKIP
- Nếu `img.width > img.height` (vẫn landscape) → browser chưa xoay → APPLY transform

```javascript
// Chỉ thêm ~6 dòng, KHÔNG hardcode gì
img.onload = function() {
    var width = img.width, height = img.height;
    
    // PTE-9503: Detect if browser already auto-applied EXIF rotation
    var browserAlreadyRotated = false;
    if (srcOrientation >= 5 && srcOrientation <= 8) {
      // Orientations 5-8 involve 90°/270° rotation
      // Raw sensor data should be landscape (width > height)
      // If image is already portrait (height > width), browser auto-rotated
      browserAlreadyRotated = (height > width);
    }
    var effectiveOrientation = browserAlreadyRotated ? 1 : srcOrientation;
    
    // ... rest of code uses effectiveOrientation instead of srcOrientation
}
```

| Tiêu chí | Đánh giá |
|----------|----------|
| **Dễ hiểu** | ✅ Logic tự giải thích — so sánh EXIF với dimension thực tế |
| **Không hardcode** | ✅ Không có magic string hay base64 |
| **Backward compatible** | ✅ Browser cũ: width > height → apply transform (đúng) |
| **Code change** | ~6 dòng trong 1 file |
| **Risk** | Rất thấp — logic rõ ràng, có thể giải thích bằng 1 câu |

---

### Solution B: Skip transform khi orientation > 1 (Đơn giản nhất)

**Ý tưởng của anh:** Nếu EXIF orientation > 1 → phone/browser đã xử lý → skip.

```javascript
// Cực kỳ đơn giản: 1 dòng
var effectiveOrientation = (srcOrientation > 1) ? 1 : srcOrientation;
```

| Tiêu chí | Đánh giá |
|----------|----------|
| **Dễ hiểu** | ✅ Cực đơn giản |
| **Backward compatible** | ❌ Browser cũ (Chrome < 81) sẽ bị sai |
| **Risk** | Trung bình — PTE app chạy WebView nên OK, nhưng nếu mở trên browser cũ thì lỗi |

## Trade-off Analysis

| Tiêu chí | Đánh giá |
|----------|----------|
| **Impact scope** | `imageFactory.js` — 1 file duy nhất. Fix cho tất cả 40+ callers (avatar, exercise, insurance, message, clinic settings) |
| **UI/UX** | Không thay đổi flow — user vẫn chọn ảnh → preview → send. Ảnh portrait hiển thị đúng chiều |
| **Performance** | Không ảnh hưởng — chỉ thêm 1 phép so sánh `height > width` (O(1)) |
| **Regression risk** | Rất thấp — ảnh orientation=1 (download) không bị ảnh hưởng vì không vào case 5-8 |
| **Testing** | Test trên Android WebView (Samsung) + Desktop Chrome + ảnh download Google |
