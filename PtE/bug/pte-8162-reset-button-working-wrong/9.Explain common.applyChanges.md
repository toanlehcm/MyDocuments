## #2. `common.applyChanges` làm gì?

```javascript
// common.js line 2151-2158
common.applyChanges = function ($scope) {
  if (!$scope) return;                              // [1] Guard: không crash nếu scope null
  var phase = $scope.$root && $scope.$root.$$phase; // [2] Lấy phase hiện tại của AngularJS digest cycle
  if (phase != '$apply' && phase != '$digest') {    // [3] Chỉ gọi $apply nếu KHÔNG đang trong digest
    $scope.$apply();                                // [4] Trigger digest cycle để update UI
  }
}
```

**Nó giải quyết 1 vấn đề kinh điển của AngularJS:**

> Khi code bên ngoài AngularJS context (callback async, Promise `.then`, `.finally`) thay đổi `$scope`, AngularJS **không biết** để re-render UI → phải gọi `$scope.$apply()` thủ công.

**Nhưng** nếu gọi `$scope.$apply()` khi đang trong digest cycle rồi → throw **`$apply already in progress`** error.

`common.applyChanges` check phase trước → **safe wrapper** tránh lỗi đó.

## #3. Tại sao cần thêm vào `.finally()`?

### Cơ chế:

```
Browser                    AngularJS Digest
   │                              │
   │  api.call() → Promise        │
   │─────────────────────────────▶│  (Angular biết)
   │                              │
   │  ... server response ...     │
   │◀─────────────────────────────│
   │                              │
   │  .then() callback chạy       │
   │  $scope.invoices = data      │  ← thay đổi scope
   │  $scope.spinnerLoading=false │
   │                              │  ← Angular CHƯA biết!
   │                              │
   │  common.applyChanges($scope) │
   │─────────────────────────────▶│  ← trigger $apply
   │                              │  ← Angular re-render UI
```

### Nếu KHÔNG có `common.applyChanges`:

```javascript
// Code cũ (trước PTE-8162):
.then((_res) => {
  $scope.getInvoiceSpinnerLoading = false;   // thay đổi scope
  // ...update invoices, columns...
},
(error) => {
  $scope.getInvoiceSpinnerLoading = false;
})
// Sau đó có:
$timeout(() => {
  $scope.$apply();   // ← cái này đảm bảo digest chạy
});
```

Code cũ dùng `$timeout(() => $scope.$apply())` để trigger digest. Điều này **hoạt động** nhưng:
- `$timeout` schedule thêm 1 macrotask mới → digest chạy **muộn hơn 1 tick**
- Nếu có exception trong `.then`, `$scope.getInvoiceSpinnerLoading` có thể không được reset (spinner treo)

### Khi thêm `common.applyChanges` vào `.finally()`:

```javascript
.then((_res) => {
  // xử lý data — KHÔNG cần tự gọi $apply
},
(error) => {
  // chỉ log error — KHÔNG cần tự gọi $apply  
})
.finally(() => {
  $scope.getInvoiceSpinnerLoading = false;   // luôn chạy, kể cả error
  common.applyChanges($scope);              // trigger digest NGAY sau finally
})
```

| Tiêu chí | Trước (cũ) | Sau (mới) |
|---|---|---|
| **Spinner luôn tắt** | ❌ Error path có thể bỏ sót | ✅ `.finally()` đảm bảo |
| **Digest timing** | Chậm 1 tick (`$timeout`) | Nhanh hơn (sync sau `.finally`) |
| **Gọi `$apply` an toàn** | `$scope.$apply()` trực tiếp → có thể throw error nếu đang trong digest | `common.applyChanges` → check phase trước, không throw |
| **Code duplication** | `$scope.getInvoiceSpinnerLoading = false` viết 2 lần (`.then` + error handler) | Viết 1 lần trong `.finally()` |

**Tóm lại**: `common.applyChanges($scope)` trong `.finally()` đảm bảo AngularJS **luôn biết** khi data đã sẵn sàng để re-render, an toàn hơn gọi `$scope.$apply()` trực tiếp, và `.finally()` đảm bảo spinner không bao giờ bị treo dù server trả error.
