# 🐛 Retrospective: Bug PTE-8162
**[RCM -> Invoice] Reset button working wrong**

Dưới đây là tổng hợp Root Cause và Solution chi tiết cho ticket PTE-8162 để anh có thể dùng làm tài liệu tham khảo hoặc thêm vào mô tả PR/Commit.

---

## 🛑 Vấn đề (Observed)
Trong màn hình RCM Invoice:
1. User chọn một Quick View.
2. Xóa cột `DOS` và click **Save**.
3. Thêm một cột bất kỳ (VD: cột X) để làm view thay đổi (`isEdited = true` → Hiện nút Reset).
4. Click nút **Reset**.
**Lỗi:** Bảng hiển thị lại cột `DOS` dù trước đó đã lưu (Save) thành công việc xóa cột này. Nút Reset đáng lẽ chỉ khôi phục về trạng thái vừa save gần nhất.

---

## 🔍 Root Cause (Nguyên nhân gốc rễ)

Có **2 lớp nguyên nhân (Root Causes)** dẫn đến lỗi này, nằm ở cơ chế quản lý state (trạng thái) của client (AngularJS) sau khi Save thành công:

### 1. `quickViewList` không được đồng bộ sau khi Save (Primary Root Cause)
Trong hàm `_saveView()`, khi API `_upsertCustomView` trả về thành công, code cũ chỉ cập nhật `view.Name = res.Name` mà **quên cập nhật** `Filters` và `DisplayColumns`. 
👉 Dẫn đến: Biến `view` trong danh sách `quickViewList` vẫn giữ mảng cột cũ (bao gồm `DOS`).

### 2. `oldView` closure bị stale (Hidden Root Cause)
Nút Reset gọi hàm `_resetView()`. Hàm này sử dụng một biến closure là `oldView` (chứa snapshot của view ở lần render panel cuối cùng).
Khi Save thành công, giao diện cập nhật nhưng **không gọi lại hàm `_renderViewToPanelFilter()`**, khiến `oldView` không được làm mới. 
👉 Dẫn đến: Khi bấm Reset, hàm `_resetView()` lấy `oldView.view` (vẫn còn chứa `DOS`) truyền vào `_processSelectQuickView(oldView.view)` để vẽ lại panel, khiến cột `DOS` hiện hồn trở lại.

---

## 🛠️ Solution (Giải pháp)

Để fix triệt để và đồng bộ với kiến trúc chuẩn của dự án (tham chiếu module ATP: `insuranceClaimManagementATP.js`), chúng ta đã thực hiện 3 thay đổi:

### 1. Fix lỗi đồng bộ `quickViewList` trong `_saveView`
Bổ sung code để đồng bộ trực tiếp response từ server vào list view hiện tại và trigger lại bảng hiển thị:
```javascript
if (view) {
  view.Name = res.Name;
  view.Filters = res.Filters; // Thêm mới
  view.DisplayColumns = res.DisplayColumns; // Thêm mới để fix lỗi hiển thị
  $scope.quickView.selectedQuickView = res._id; // Giữ lại view đang chọn
}
// Đảm bảo luôn apply view sau khi save (nằm trong khối .finally)
_applyView(false); 
```

### 2. Refactor `_resetView` gọi API lấy data mới nhất (Theo chuẩn ATP)
Thay vì phụ thuộc vào biến `oldView` dễ bị stale, ta đổi `_resetView` để gọi API lấy thẳng data từ DB giống hệt cách ATP đang làm.
- **Thêm hàm mới `_getView(viewId)`:** Gọi API `v4GetCustomViewDetail`.
- **Sửa `_resetView`:** 
```javascript
// Đổi từ: _processSelectQuickView(oldView.view);
// Thành:
_getView($scope.searchTerms.viewId).then((res) => {
    if (res) {
      $scope.searchTerms.isDefaultOfSystem = res.IsDefaultOfSystem;
      oldView.view._id = res.viewId;
      _processSelectQuickView(oldView.view, res); // Truyền response server vào hàm
      _applyView(false);
      // ... showToast
    }
});
```

### 3. Nâng cấp `_processSelectQuickView` nhận param thứ 2
Để hàm `_processSelectQuickView` có thể ưu tiên dùng data mới từ server thay vì tìm kiếm trong `quickViewList` (có nguy cơ bị stale), ta thêm param `viewData` (optional) để tương thích ngược (backward compatible) với các hàm cũ:
```javascript
function _processSelectQuickView(view, viewData) {
  let foundView;
  if (viewData) {
    foundView = viewData; // Bypass quickViewList, dùng server data
  } else {
    foundView = $scope.quickView.quickViewList.find((v) => v._id === view._id);
  }
  // ... render panel
}
```

> [!NOTE]
> **Kết quả:** Nút Reset giờ đây luôn khôi phục view về trạng thái chính xác nhất trên server (tương đương với một hard-sync), giải quyết triệt để lỗi hiển thị sai cột hoặc bộ lọc sau khi lưu. Code được đồng nhất 100% về mặt pattern với `insuranceClaimManagementATP.js`.
