# Bug Retrospective: PTE-8162
**[RCM -> Invoice] Reset button working wrong**

## Issue Description
In the RCM Invoice tab, the Reset button within the Quick View feature failed to restore the most recently saved state. Specifically, after a user removed a display column (such as `DOS`), saved the view, and subsequently modified the view again to trigger the Reset button, clicking Reset incorrectly restored the previously deleted column instead of reverting to the recently saved configuration.

## Root Cause Analysis
The issue stemmed from two related state-management synchronization problems on the client side:

1. **Stale Local State in `quickViewList`**:
   Upon a successful save action, the `_saveView` function updated the view's name locally but omitted updating the `Filters` and `DisplayColumns` properties with the response from the server. Consequently, the local client state retained the outdated column list.

2. **Stale Closure Reference (`oldView`)**:
   The `_resetView` function relied on a closure variable named `oldView`, which captures a snapshot of the view's layout. Because the application did not refresh this snapshot after a successful save, `oldView` remained stale. When the Reset button was clicked, the function used this outdated snapshot to re-render the interface, causing the deleted columns to reappear.

## Implemented Solution
To resolve the synchronization issues and align the code with the established architecture patterns used elsewhere in the application (specifically in the ATP module), the following updates were made:

1. **Synchronized Local State on Save**:
   Updated the `_saveView` function to immediately map the `Filters` and `DisplayColumns` from the server response back into the local view object. This ensures the local memory accurately reflects the database state.

2. **Implemented Server-Side Data Fetch on Reset**:
   Refactored the `_resetView` function to abandon its reliance on the stale `oldView` snapshot. A new function, `_getView`, was introduced to make an API call to the backend. The `_resetView` function now fetches the latest, accurate view data directly from the server before triggering a re-render.

3. **Enhanced View Processing Function**:
   Modified the `_processSelectQuickView` function to accept an optional second parameter containing the fresh server response data. This allows the rendering logic to bypass the potentially outdated local list and directly utilize the most accurate data.
