# Refactor: Tái sử dụng logic Print PDF cho Patient Dashboard

## Bối cảnh

Hiện tại có **2 nơi** xử lý logic in PDF, và chúng **duplicate nhau** nhưng có chất lượng khác nhau:

| | `claimsPatientATP.js` (Patient Dashboard) | `commonService.js` (Claim Management) |
|---|---|---|
| **Hàm** | `print1500(url, claimId)` — dòng 3264 | `_printPdf(pdfUrl)` — dòng 1040 |
| **WEB** | ✅ `printJS(...)` | ✅ `printJS(...)` |
| **Android** | ❌ Gọi thẳng `$cordovaPrinter.print(url)` | ✅ `$cordovaPrinter.print(pdfUrl)` |
| **iOS** | ❌ Gọi thẳng `$cordovaPrinter.print(url)` — **KHÔNG CÓ download trước** | ✅ Download file trước → rồi mới `$cordovaPrinter.print(fileURI)` |
| **Guard** | ❌ Không kiểm tra `pdfUrl` null/undefined | ✅ Guard `if (!pdfUrl) return` |
| **Error handling** | ❌ Không có | ✅ Có `try/catch` cho iOS |

> [!CAUTION]
> `print1500` trong `claimsPatientATP.js` đang **thiếu logic download trước khi print trên iOS**, dẫn đến **print không hoạt động trên iOS** vì iOS không cho phép `$cordovaPrinter.print()` trực tiếp từ URL AWS S3.

## Phân tích 3 Solutions

### Solution A: Gọi thẳng `commonService.printPdf()` (Expose `_printPdf`)

**Cách làm:** Thêm `this.printPdf = _printPdf` vào public API của `commonService`, rồi trong `print1500` gọi `commonService.printPdf(url)`.

**Code sau khi sửa (claimsPatientATP.js):**
```javascript
function print1500(url, claimId) {
  commonService.printPdf(url)
  _updateStatusAndPrintInfo(claimId).then(_res => {
    loadListClaim($scope.pageIndexInsuranceClaim, $scope.pageSizeInsuranceClaim)
  })
}
```

| Ưu điểm | Nhược điểm |
|---|---|
| ✅ Xóa sạch duplicate code — `print1500` chỉ còn 3-4 dòng | ⚠️ `_updateStatusAndPrintInfo` nằm ngoài `_printPdf`, timing có thể lệch (print async nhưng updateStatus chạy song song) |
| ✅ Fix luôn bug iOS (vì `_printPdf` đã có logic download trước) | ⚠️ Cần thêm 1 dòng expose vào commonService |
| ✅ Maintain dễ: sửa 1 chỗ, đồng bộ tất cả | |
| ✅ Tuân thủ pattern: logic dùng chung nằm ở Service (theo `.agent/code-standards/patterns.md`) | |

**Rủi ro:** Thấp. `_printPdf` đã được test kỹ qua Claim Management flow. Chỉ cần xác nhận rằng `_updateStatusAndPrintInfo` không cần chờ print xong mới chạy (hiện tại nó cũng chạy song song rồi).

### Solution B: Copy logic iOS/Android từ `_printPdf` vào `print1500`

**Cách làm:** Giữ nguyên `print1500` là hàm local trong controller, nhưng bổ sung logic phân nhánh Android/iOS giống `_printPdf`.

**Code sau khi sửa (claimsPatientATP.js):**
```javascript
function print1500(url, claimId) {
  if (!url) return
  if ($rootScope.appType === AppTypeConst.WEB) {
    printJS({ printable: url, type: 'pdf', showModal: true })
  } else if ($rootScope.DeviceType === 'android') {
    $cordovaPrinter.print(url)
  } else {
    // iOS: download first, then print
    var fileName = _extractFileNameFromUrl(url) // phải copy hàm này vào
    downloadService.download(url, fileName).then(function(fileURI) {
      if (fileURI) {
        $cordovaPrinter.print(fileURI, { name: 'Insurance_Claim' })
      }
    })
  }
  _updateStatusAndPrintInfo(claimId).then(_res => {
    loadListClaim($scope.pageIndexInsuranceClaim, $scope.pageSizeInsuranceClaim)
  })
}
```

| Ưu điểm | Nhược điểm |
|---|---|
| ✅ Không cần sửa commonService | ❌ **Duplicate code** — vi phạm DRY |
| ✅ Có thể tùy chỉnh riêng cho Patient Dashboard | ❌ Cần copy thêm `_extractFileNameFromUrl` hoặc hard-code filename |
| | ❌ Sửa bug ở đây phải nhớ sửa ở commonService nữa (và ngược lại) |
| | ❌ Vi phạm pattern: business logic trong Controller (anti-pattern theo `.agent/code-standards/anti-patterns.md`) |

**Rủi ro:** Trung bình. Code chạy đúng nhưng **nợ kỹ thuật tăng** (maintenance cost cao về sau).

| Ưu điểm | Nhược điểm |

| ✅ Xóa sạch duplicate | ❌ `announceClick` trong `claimsPatientATP.js` nhận `url` (đã resolved), còn `_printCMS1500` nhận `claim` object (tự gọi API lấy URL) → **chạy API lấy URL 2 lần** |

**Rủi ro:** Cao. Refactor nhiều file, thay đổi flow data. Cần test regression cả 2 màn hình.
