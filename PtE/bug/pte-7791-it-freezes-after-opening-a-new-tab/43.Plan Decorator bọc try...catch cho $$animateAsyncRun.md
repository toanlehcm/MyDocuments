# Plan: Decorator bọc try...catch cho $$animateAsyncRun

## 1. Tại sao try...catch trong showPopupAddEditTask KHÔNG FIX được Root Cause?

### Vấn đề gốc trong `angular.js` (KHÔNG SỬA FILE NÀY)
```javascript
// D:\Sources\pteverywhere\Client\bower_components\angular\angular.js
$$rAF(function() {
    for (var i = 0; i < waitQueue.length; i++) {
        waitQueue[i]();  // ← 1 callback throw → vòng for dừng
    }
    waitQueue = [];      // ← KHÔNG BAO GIỜ chạy → queue kẹt vĩnh viễn
});
```

### Tại sao try...catch trong showPopupAddEditTask bất lực?

```
                    ┌─────────────────────────┐
                    │   JavaScript Engine      │
                    │                          │
  Tick 1 (sync)     │   Tick 2 (async - rAF)   │
  ┌──────────────┐  │  ┌─────────────────────┐ │
  │ try {        │  │  │ $$rAF(function() {   │ │
  │   testTick() │──┼──│   waitQueue[0]()  ← CRASH │
  │ } catch {    │  │  │   waitQueue = [] ← SKIP   │
  │   // ĐÃ THOÁT│  │  │ })                  │ │
  │ }            │  │  │                      │ │
  └──────────────┘  │  └─────────────────────┘ │
                    └─────────────────────────┘
         ↑ catch ĐÃ THOÁT         ↑ lỗi NỔ Ở ĐÂY
         trước khi rAF chạy       trên call stack KHÁC
```

**Kết luận:** Bất kỳ `try...catch` nào đặt trong `showPopupAddEditTask` đều chạy ĐỒNG BỘ và thoát ra TRƯỚC KHI `requestAnimationFrame` callback được gọi. Chúng nằm trên 2 call stack hoàn toàn độc lập. Không có cách nào `try...catch` ở Tick 1 bắt được lỗi xảy ra ở Tick 2.

## 2. Phương Án Thực Sự Khả Thi: `$provide.decorator`

### Decorator là gì?
AngularJS cung cấp cơ chế `$provide.decorator` cho phép **bọc thêm logic xung quanh một service** mà KHÔNG SỬA FILE SOURCE của service đó. Đây là cơ chế chính thức của AngularJS, không phải hack.

### Nguyên lý

Tạo một file mới (ví dụ `safeAnimateAsyncRun.js`) trong folder `app/scripts/decorator/`. File này sẽ:
1. Dùng `$provide.decorator('$$animateAsyncRun', ...)` để can thiệp vào service `$$animateAsyncRun` tại thời điểm config
2. Thay thế hàm `waitForTick` bên trong bằng phiên bản có `try...catch` quanh mỗi callback
3. Đảm bảo `waitQueue = []` LUÔN LUÔN được gọi kể cả khi 1 callback throw

### Code đề xuất

#### [NEW] [safeAnimateAsyncRun.js](file:///d:/Sources/pteverywhere/Client/app/scripts/decorator/safeAnimateAsyncRun.js)

```javascript
'use strict'

// Workaround: Patch $$animateAsyncRun to prevent waitQueue deadlock.
// Root cause: In angular.js's $$AnimateAsyncRunFactoryProvider, the flush
// loop has no try...catch. If ONE callback throws, waitQueue = [] never
// runs → queue stuck forever → all subsequent animations deadlock.
//
// This decorator wraps each callback in try...catch so the loop always
// completes and waitQueue is always reset.
// NOTE: This does NOT modify angular.js source file.
angular.module('PtEMobile').config(['$provide', function($provide) {
  $provide.decorator('$$animateAsyncRun', ['$delegate', '$$rAF', function($delegate, $$rAF) {
    // Replacement: safe version of $$AnimateAsyncRunFactory
    var waitQueue = []

    function waitForTick(fn) {
      waitQueue.push(fn)
      if (waitQueue.length > 1) return
      $$rAF(function() {
        // Snapshot and reset FIRST, then execute.
        // This guarantees waitQueue is always cleared even if a callback throws.
        var queue = waitQueue
        waitQueue = []
        for (var i = 0; i < queue.length; i++) {
          try {
            queue[i]()
          } catch (e) {
            // Swallow error from individual callback to protect the rest of the queue.
            // Log for debugging but do NOT re-throw.
          }
        }
      })
    }

    return function() {
      var passed = false
      waitForTick(function() {
        passed = true
      })
      return function(callback) {
        if (passed) {
          callback()
        } else {
          waitForTick(callback)
        }
      }
    }
  }])
}])
```

## 4. Phản Biện (Anh yêu cầu em challenge)

### ⚠️ Rủi ro 1: Nuốt lỗi im lặng (Error Swallowing)

Decorator dùng `catch (e) {}` rỗng → Lỗi bị nuốt mất, developer không biết có bug ẩn.

**Đánh giá:** Rủi ro THỰC SỰ CÓ nhưng ở mức CÓ THỂ CHẤP NHẬN. Lý do:
- Lỗi trong animation callback thường là lỗi phụ (class manipulation, style calculation) chứ không phải lỗi nghiệp vụ
- Nếu thêm `console.error` vào catch → developer vẫn thấy lỗi ở DevTools Console nhưng app không chết
- So sánh: Hậu quả của nuốt lỗi (1 animation bị skip) vs hậu quả của không nuốt (TOÀN BỘ APP ĐÓNG BĂNG) → Nuốt lỗi rõ ràng tốt hơn

**Khuyến nghị:** Thêm `console.error` vào catch block thay vì để trống.

### ⚠️ Rủi ro 2: Decorator có đúng là "không sửa Core" không?

**Phản biện nghiêm túc:** `$provide.decorator` KHÔNG sửa file `angular.js`. Nó hoạt động ở Runtime, tại thời điểm AngularJS khởi tạo DI Container. Đây là cơ chế chính thức được AngularJS thiết kế cho mục đích này (xem [AngularJS $provide.decorator docs](https://docs.angularjs.org/api/auto/service/$provide#decorator)).

Tuy nhiên, Decorator THAY THẾ HOÀN TOÀN implementation của `$$animateAsyncRun`. Nếu AngularJS Team (giả sử) cập nhật logic bên trong `$$AnimateAsyncRunFactoryProvider`, Decorator của chúng ta sẽ ghi đè mất phiên bản mới.

**Đánh giá:** Rủi ro NÀY KHÔNG TỒN TẠI trong thực tế vì AngularJS 1.x đã End-of-Life từ 2021. Sẽ không có bản cập nhật nào nữa.

### ⚠️ Rủi ro 3: Decorator có thể không nhận được `$$rAF` đúng

`$$rAF` là internal service. Decorator inject `$$rAF` từ DI container — đây là cùng instance mà `angular.js` sử dụng. Tuy nhiên, nếu có Decorator KHÁC cũng can thiệp `$$rAF`, thứ tự decorator có thể gây conflict.

**Đánh giá:** Em đã grep toàn bộ codebase, chưa thấy decorator nào cho `$$rAF`. Rủi ro rất thấp.

### ⚠️ Rủi ro 4: Snapshot-then-reset pattern khác với original

Code gốc:
```javascript
for (var i = 0; i < waitQueue.length; i++) { waitQueue[i]() }
waitQueue = []  // reset SAU khi loop xong
```

Code decorator:
```javascript
var queue = waitQueue
waitQueue = []  // reset TRƯỚC khi loop chạy
for (var i = 0; i < queue.length; i++) { try { queue[i]() } catch(e) {} }
```

**Khác biệt:** Nếu một callback bên trong `queue[i]()` tạo thêm animation MỚI (gọi `waitForTick` lần nữa), thì:
- **Code gốc:** Animation mới được push vào `waitQueue` hiện tại, nhưng vì loop đang chạy trên `waitQueue`, item mới SẼ ĐƯỢC xử lý ngay trong lần flush hiện tại (vì `waitQueue.length` tăng lên)
- **Code decorator:** Animation mới được push vào `waitQueue` mới (đã reset = []), sẽ được xử lý ở lần `$$rAF` TIẾP THEO

**Đánh giá:** Đây là khác biệt nhỏ về timing (1 frame = ~16ms chậm hơn cho nested animation). Trong thực tế, hiếm khi một animation callback tạo ra animation MỚI ngay lập tức. Và nếu có, chậm 16ms là KHÔNG ĐO ĐƯỢC bằng mắt thường. Trade-off này hoàn toàn chấp nhận được để đổi lấy sự an toàn.
