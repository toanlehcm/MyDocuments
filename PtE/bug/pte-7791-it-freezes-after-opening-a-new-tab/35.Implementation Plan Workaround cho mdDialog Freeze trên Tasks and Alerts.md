# Implementation Plan: Workaround cho $mdDialog Freeze trên Tasks & Alerts

## 1. ROOT CAUSE — Giải thích cho Leader

### 1.1. Vấn đề gì xảy ra?

Khi user **F5 (refresh) trang Schedule → click nút Associated Task → Tab Tasks & Alerts mở ra → click "+Task"**, dialog hiện lên nhưng **đóng băng hoàn toàn** — không thể nhập liệu, không thể đóng. `onShowing` callback được gọi, nhưng `onComplete` **không bao giờ được gọi**.

### 1.2. Root Cause nằm ở đâu?

**File:** `bower_components/angular/angular.js`  
**Dòng:** 6244–6258  
**Component:** `$$AnimateAsyncRunFactoryProvider` — một internal service của AngularJS core dùng chung cho **tất cả animation callbacks** trong app.

```javascript
// angular.js line 6244-6258
var $$AnimateAsyncRunFactoryProvider = function() {
  this.$get = ['$$rAF', function($$rAF) {
    var waitQueue = [];                              // ← Singleton shared queue

    function waitForTick(fn) {
      waitQueue.push(fn);
      if (waitQueue.length > 1) return;              // ← Piggyback, không schedule $$rAF mới
      $$rAF(function() {
        for (var i = 0; i < waitQueue.length; i++) {
          waitQueue[i]();                             // ← KHÔNG CÓ try/catch
        }
        waitQueue = [];                              // ← Dòng này KHÔNG BAO GIỜ chạy nếu throw
      });
    }
    // ...
  }];
};
```

### 1.3. Tại sao nó là Root Cause? (Chuỗi chứng minh)

**Cơ chế lỗi:**

1. `waitQueue` là **singleton** — chỉ có **1 instance duy nhất** cho toàn bộ app.
2. Khi flush queue, vòng lặp `for` gọi `waitQueue[i]()` **không có `try/catch`**.
3. Nếu **bất kỳ 1 callback nào throw error**, vòng lặp dừng ngay → `waitQueue = []` (dòng reset) **không bao giờ chạy**.
4. Từ thời điểm đó, `waitQueue.length` luôn `> 0` → mọi callback mới chỉ `push` vào queue mà **không bao giờ schedule `$$rAF` mới** (do điều kiện `if (waitQueue.length > 1) return`).
5. Kết quả: **Tất cả animation callback trong app vĩnh viễn bị orphaned** — bao gồm callback `complete()` của `$mdDialog`.

**Khi nào error throw?**

- Khi F5 trang Schedule → có **double route change** (`/tasks/.../Appointment/...` → `/tasks`).
- Double route change gây ra 2x `TasksManagementCtrl` instantiation.
- Animation callbacks từ view destruction của route đầu tiên throw error khi chạy trong `waitQueue` flush loop (vì DOM element đã bị remove).
- Lúc này queue bị stuck **vĩnh viễn**.

## 2. PHƯƠNG ÁN WORKAROUND (Không sửa Angular core)

### Nguyên tắc: Vì `waitQueue` bị stuck → animation callback không fire → `onComplete` không fire → `init()` không được gọi → dialog freeze. Workaround cần **bypass animation path** hoặc **không phụ thuộc vào `onComplete`**.

---

### Phương án A: Bypass `onComplete` — Dùng `onShowing` + `$timeout` thay thế (KHUYẾN NGHỊ)

**Ý tưởng:** Không đợi `onComplete` (vì nó phụ thuộc animation). Thay vào đó, gọi `init()` trực tiếp từ `onShowing` với `$timeout` để defer sang next digest cycle.

**Thay đổi trong `app.js` — hàm `showPopupAddEditTask()`:**

```javascript
$mdDialog.show({
    // ... existing config ...
    onShowing: function(scope, element) {
        // Workaround: gọi init từ onShowing thay vì đợi onComplete
        // Dùng $timeout để defer sang next tick, đảm bảo DOM đã ready
        var _$timeout = angular.element(document.body).injector().get('$timeout');
        _$timeout(function() {
            scope.dialogReady = true;
            if (scope.$initFromDialog) {
                scope.$initFromDialog();
            }
        }, 0);
    },
    onComplete: function(scope) {
        // Giữ lại như safety net — nếu animation hoạt động bình thường,
        // check flag để không gọi init() 2 lần
        if (!scope.dialogReady) {
            scope.dialogReady = true;
            if (scope.$initFromDialog) {
                scope.$initFromDialog();
            }
        }
    },
    // ...
});
```

**Thay đổi trong `popupAddEditTaskPtE.js` — hàm `init()`:**

```javascript
function init() {
    // Guard: chống gọi 2 lần (onShowing $timeout + onComplete nếu cả 2 đều fire)
    if ($scope._initCalled) return;
    $scope._initCalled = true;
    
    // ... code init hiện tại giữ nguyên ...
}
```

**Trade-off:**

| Tiêu chí | Đánh giá |
|----------|----------|
| **UX** | ✅ Gần như không thay đổi. `$timeout(fn, 0)` chỉ delay ~4-16ms (1 frame). User không nhận thấy. |
| **Time Complexity** | ✅ Thay đổi rất nhỏ (~15 phút code + test). Chỉ sửa 2 file. |
| **Risk** | ⚠️ Thấp. `onShowing` fire khi dialog DOM đã được insert vào document nhưng animation chưa complete. Với `$timeout(0)` defer, DOM đã sẵn sàng cho `init()`. Cần kiểm tra các field có `ng-if="initialized"` vẫn render đúng. |
| **Side effects** | ⚠️ `init()` chạy khi animation đang diễn ra thay vì sau animation. Với dialog này (form đơn giản, ~176 DOM nodes) không gây vấn đề. |
| **Guard clause** | ✅ `_initCalled` flag đảm bảo init chỉ chạy 1 lần dù cả 2 path đều fire (trường hợp animation không stuck). |

### Phương án B: Disable Animation cho dialog cụ thể này

**Ý tưởng:** Tắt animation cho `$mdDialog` này → Angular skip toàn bộ animation path → `onComplete` fire đồng bộ → không phụ thuộc `waitQueue`.

**Thay đổi trong `app.js`:**

```javascript
$mdDialog.show({
    // ... existing config ...
    onShowing: function(scope, element) {
        // Disable animation cho dialog element
        var _$animate = angular.element(document.body).injector().get('$animate');
        _$animate.enabled(element, false);
        if (element.parent()) {
            _$animate.enabled(element.parent(), false);
        }
        // Force remove animation classes
        element.css({ 'transition': 'none', 'animation': 'none' });
        if (element.parent()) {
            element.parent().css({ 'transition': 'none', 'animation': 'none' });
        }
    },
    onComplete: function(scope) {
        // Vẫn giữ nguyên
        scope.dialogReady = true;
        if (scope.$initFromDialog) {
            scope.$initFromDialog();
        }
    },
    // ...
});
```

**Trade-off:**

| Tiêu chí | Đánh giá |
|----------|----------|
| **UX** | ⚠️ Dialog mở **không có animation** (không slide-in, không fade-in). Cảm giác "cứng" hơn. |
| **Time Complexity** | ✅ Rất nhỏ (~10 phút). Chỉ sửa 1 file. |
| **Risk** | ⚠️ Trung bình. `$animate.enabled(element, false)` có thể KHÔNG đủ vì `$$interimElement` (internal của `$mdDialog`) đã schedule animation TRƯỚC khi `onShowing` fire. Nếu animation đã được queue trước đó, disable ở `onShowing` có thể quá muộn. **Cần test kỹ.** |
| **Side effects** | ⚠️ Mất hiệu ứng mở/đóng dialog. Chỉ ảnh hưởng popup Task này. |

### Phương án C: Decorator cho `$$animateAsyncRun` — thêm try/catch (Recommended if scope allows)

**Ý tưởng:** Dùng Angular `$provide.decorator` để wrap `$$animateAsyncRun` với try/catch, **không sửa angular.js source**.

**Thay đổi trong `app.js` — phần `module.config()`:**

```javascript
angular.module('PtEMobile').config(['$provide', function($provide) {
    $provide.decorator('$$animateAsyncRun', ['$delegate', '$$rAF', function($delegate, $$rAF) {
        // Không dùng $delegate vì cần thay thế hoàn toàn logic waitForTick
        var waitQueue = [];

        function waitForTick(fn) {
            waitQueue.push(fn);
            if (waitQueue.length > 1) return;
            $$rAF(function() {
                var queue = waitQueue;
                waitQueue = [];  // Reset TRƯỚC khi chạy — đảm bảo queue mới luôn clean
                for (var i = 0; i < queue.length; i++) {
                    try {
                        queue[i]();
                    } catch (e) {
                        // Log error nhưng không dừng vòng lặp
                        console.error('$$animateAsyncRun callback error:', e);
                    }
                }
            });
        }

        return function() {
            var passed = false;
            waitForTick(function() {
                passed = true;
            });
            return function(callback) {
                if (passed) {
                    callback();
                } else {
                    waitForTick(callback);
                }
            };
        };
    }]);
}]);
```

**Trade-off:**

| Tiêu chí | Đánh giá |
|----------|----------|
| **UX** | ✅ **Không thay đổi** gì. Animation vẫn chạy bình thường. Dialog vẫn có hiệu ứng slide-in. |
| **Time Complexity** | ✅ Nhỏ (~20-30 phút code + test). Chỉ thêm 1 block code trong config. |
| **Risk** | ⚠️ Trung bình. Decorator thay thế hoàn toàn logic `waitForTick`. Cần test regression trên các dialog/animation khác. Tuy nhiên logic gần như **giống hệt** Angular core, chỉ thêm `try/catch` + đổi thứ tự reset `waitQueue = []` lên trước vòng lặp. |
| **Side effects** | ✅ Sửa root cause ở application level → **tất cả dialog/animation đều được bảo vệ**, không chỉ popup Task. |
| **Lưu ý** | Decorator chạy ở `config` phase → chỉ apply 1 lần khi app bootstrap. Nếu ngAnimate cũng register service này, cần kiểm thứ tự load (đã xác nhận ngAnimate **KHÔNG override** `$$animateAsyncRun`). |

## 4. KHUYẾN NGHỊ

### Phương án tối ưu: **A + C kết hợp**

1. **Áp dụng C (Decorator)** — fix root cause ở application level, bảo vệ toàn bộ app khỏi `waitQueue` stuck cho mọi dialog/animation.
