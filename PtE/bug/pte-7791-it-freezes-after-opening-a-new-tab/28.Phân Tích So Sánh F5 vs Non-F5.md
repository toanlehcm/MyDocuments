## 2. ĐIỂM KHÁC NHAU (Tab 1 — Calendar page)

> [!CAUTION]
> Điểm khác biệt **DUY NHẤT** nằm ở **Tab 1 (Calendar page)**, KHÔNG phải Tab 2.

### 2.1 `[PAGE START]` — KHÁC ⚡

| Metric | Case A (F5) | Case B (Non-F5) |
|--------|-------------|------------------|
| `🟠 [PAGE START]` trên Tab 1 | ✅ CÓ — `[PAGE START] /` | ❌ KHÔNG CÓ |
| Vị trí | Case A hình 1, dòng đầu console | Case B hình 1, KHÔNG có dòng này |

**Giải thích:**
- `[PAGE START]` nằm trong `app.run()` → chỉ chạy khi Angular **bootstrap lần đầu**

### 2.2 `destroyScheduler` — KHÁC ⚡

| Metric | Case A (F5) | Case B (Non-F5) |

**Giải thích:**
- **Case A (F5):** F5 = browser xóa sạch memory → Scheduler đã bị xóa ở level browser, không cần Angular gọi `destroyScheduler`
- **Case B (Non-F5):** SPA navigation → Angular phải tự cleanup scheduler trước khi tạo lại → gọi `destroyScheduler` → cleanup có tổ chức, ít tốn resource hơn

## 3. TẠI SAO CÓ SỰ GIỐNG VÀ KHÁC NÀY

### Tại sao Tab 2 GIỐNG nhau?

Tab 2 luôn được mở bằng `window.open(url, '_blank', 'noopener,noreferrer')`.

Dù có F5 hay không, Tab 2 **luôn phải:**
- Load lại ALL JavaScript files (AngularJS, Material, lodash, moment...)
- Bootstrap Angular từ đầu
- Gọi `v3GetDataSession` để build lại `$rootScope`
- Gọi tất cả API (v4GetPublicSystemConfig, v4GetAllStateByCountryATP...)
- Kết nối WebSocket mới (và bị fail)

→ **Tab 2 không biết và không quan tâm** Tab 1 có F5 hay không. Tab 2 luôn làm mọi thứ giống nhau.

### Tại sao Tab 1 KHÁC nhau?

| Khía cạnh | F5 (Case A) | Non-F5 (Case B) |
|-----------|-------------|------------------|
| **JS parsing** | Re-parse ALL 50+ JS files (~5MB) | Đã parse, JIT-compiled, trong V8 memory |
| **CSS parsing** | Re-parse ALL CSS + SCSS | Đã computed, trong render tree |
| **Template cache** | Empty → compile mọi template | Cached từ session trước |
| **Angular bootstrap** | Full bootstrap (app.run, config, services) | Đã bootstrap, chỉ route change |
| **dhtmlx Scheduler** | Build mới 100% từ scratch | destroyScheduler → rebuild (có cached resources) |
| **Appointments data** | Fetch lại từ API | Fetch lại từ API (nhưng template đã compiled) |
| **Total CPU time** | **NẶNG — 10-30 giây** | **NHẸ — 2-5 giây** |

### 🔬 3 câu hỏi Leader có thể hỏi + câu trả lời

**Q1: "Sao không fix được?"**
> A: Đây là giới hạn của kiến trúc SPA legacy (AngularJS 1.x) + Chrome renderer process model. Cả 2 tabs cùng domain luôn chia sẻ 1 main thread. Muốn fix triệt để phải migrate sang framework hiện đại (React/Angular 2+) hỗ trợ Web Workers, hoặc chuyển sang micro-frontend architecture.

**Q2: "Có cách nào giảm thiểu không?"**
> A: Có 2 giải pháp nhẹ:
> 1. Thêm "loading" state cho popup — dropdown hiện skeleton thay vì freeze
> 2. Mở Tab 2 trì hoãn 1-2 giây sau khi click — cho Tab 1 thêm thời gian settle

**Q3: "Tần suất user gặp bug này?"**
> A: Rất thấp. User hiếm khi nhấn F5 (workflow bình thường là mở app 1 lần rồi dùng cả ngày). Và ngay cả khi nhấn F5, phải mở Tab 2 + click popup + click dropdown **NGAY LẬP TỨC** (trong vài giây đầu) mới bị.

## 5. CHI TIẾT BỔ SUNG QUAN TRỌNG

### 5.1 `safeHtml Auto-payment failed` (Hình 4 Non-F5)

Trong Case B hình 4, có **hàng loạt** dòng:
```
safeHtml Auto-payment failed: Invoice 1774496243891 of Patient 151, 
eResponseMessage c/b unknown, response 7080
```

Đây là notifications từ server qua WebSocket, KHÔNG liên quan đến freeze. Nhưng chúng cho thấy server đang **gửi nhiều event cùng lúc** → nếu mạng chậm, các event này có thể chiếm bandwidth và delay các API response khác.

### 5.2 Tại sao WebSocket FAILED ở cả 2 case?

```
WebSocket connection to 'wss://...' failed:
WebSocket is closed before the connection is established.
```

Cả 2 case đều fail vì Tab 2 mở bằng `noopener,noreferrer` → **không kế thừa** WebSocket connection từ Tab 1. Tab 2 phải tạo WebSocket connection mới. Trong quá trình setup, page đang tải JS → WebSocket chưa kịp handshake → bị close.

Nhưng **sau đó cả 2 case đều reconnect thành công** (dòng `connected!` trong hình 4). Nên WebSocket failure KHÔNG phải nguyên nhân freeze — nó chỉ là artifact của việc mở tab mới.

### 5.3 Chrome's Same-Site Renderer Process Sharing

Từ Chrome 67+, Chrome sử dụng **Site Isolation** — mỗi site (domain) có 1 renderer process. Khi mở Tab 2 cùng domain `localhost:9002`:
- Cả 2 tabs chia sẻ 1 renderer process
- 1 renderer process = 1 main thread
- AngularJS là 100% single-threaded (no Web Workers)
- → 2 tabs tranh giành CPU qua 1 main thread

Đây là giới hạn kiến trúc, không phải bug code.

### 5.4 Wait Test — Cách chứng minh nhanh nhất

Nếu leader muốn bằng chứng đơn giản nhất:

1. **F5 → Login → Calendar → ĐỢI 30 giây → View Tasks → Popup → Click dropdown → OK ✅**
2. **F5 → Login → Calendar → NGAY LẬP TỨC View Tasks → Popup → Click dropdown → FREEZE ❌**

Sự khác biệt DUY NHẤT giữa 2 lần test = **thời gian đợi**. Nếu đợi đủ lâu cho Tab 1 xong → không freeze. Chứng minh đây là timing issue.
