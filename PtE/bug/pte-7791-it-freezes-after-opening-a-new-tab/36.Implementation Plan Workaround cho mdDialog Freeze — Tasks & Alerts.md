# Implementation Plan: Workaround cho $mdDialog Freeze — Tasks & Alerts

**Phương án:** Workaround ở cấp View/Controller. **TUYỆT ĐỐI KHÔNG** sửa source Angular core.

---

## 1. ROOT CAUSE — Giải thích cho Leader

### 1.1. Vấn đề gì xảy ra?

Khi user **F5 trang Schedule → click Associated Task → Tab Tasks & Alerts mở ra → click "+Task"**, dialog popup hiện lên nhưng **đóng băng hoàn toàn** — không thể nhập liệu, không thể đóng.

### 1.2. Root Cause nằm ở đâu?

**File:** `bower_components/angular/angular.js`  
**Dòng:** 6244–6258  
**Component:** `$$AnimateAsyncRunFactoryProvider` — internal service của AngularJS core, quản lý **hàng đợi callback animation** (waitQueue) cho toàn bộ app.

```javascript
// angular.js line 6244-6258 — ĐÂY LÀ ROOT CAUSE
var $$AnimateAsyncRunFactoryProvider = function() {
  this.$get = ['$$rAF', function($$rAF) {
    var waitQueue = [];                              // ← SINGLETON — 1 queue duy nhất cho cả app

    function waitForTick(fn) {
      waitQueue.push(fn);
      if (waitQueue.length > 1) return;              // ← Nếu queue đã có item, chỉ push thêm, KHÔNG schedule $$rAF mới
      $$rAF(function() {                             // ← Chỉ schedule khi queue từ 0→1
        for (var i = 0; i < waitQueue.length; i++) {
          waitQueue[i]();                             // ← KHÔNG CÓ try/catch — crash 1 callback = chết cả queue
        }
        waitQueue = [];                              // ← DÒNG NÀY KHÔNG BAO GIỜ CHẠY nếu dòng trên throw error
      });
    }
    // ...
  }];
};
```

### 1.3. Giải thích cơ chế lỗi (cho leader dễ hiểu)

**Ví dụ đơn giản:** Tưởng tượng `waitQueue` là **1 chiếc xe bus duy nhất** cho toàn bộ app.

1. Nhiều "animation callback" (hành khách) xếp hàng lên xe bus.
2. Xe bus chuyến đầu khởi hành, đi đón hành khách (gọi `waitQueue[i]()` lần lượt).
3. **1 hành khách (callback) gây sự cố** (throw error) → xe bus dừng giữa đường → **không bao giờ quay về bến** (`waitQueue = []` không bao giờ chạy).
4. Các hành khách mới tiếp tục xếp hàng, nhưng vì xe bus chưa "quay về bến" nên không ai gọi chuyến xe mới (`waitQueue.length > 1 → return`).
5. **Kết quả:** Tất cả animation callback trong app bị mắc kẹt vĩnh viễn.

**Chuỗi sự kiện cụ thể:**

```
F5 trang Schedule
  → User click "Associated Task" button
  → Browser mở tab mới: /tasks/:patientId/:relatedTo/:id
  → Angular route match → TasksManagementCtrl khởi tạo LẦN 1, view render + animation bắt đầu
  → Controller xử lý params → redirect sang /tasks (double route change)
  → TasksManagementCtrl khởi tạo LẦN 2, view LẦN 1 bị destroy + animation cleanup
  → ⚡ Animation callback từ view destruction throw error trong waitQueue flush loop
  → waitQueue STUCK VĨNH VIỄN
  → User click "+Task" → $mdDialog.show() → dialog hiện lên
  → Animation completion callback push vào waitQueue STUCK → KHÔNG BAO GIỜ FIRE
  → onComplete KHÔNG được gọi → init() KHÔNG chạy → dialog freeze
  → $mdDialog.hide() cũng dùng animation → cũng STUCK → không đóng được dialog
```

### 1.4. Tại sao biết nó là root cause? (Bằng chứng)

Đã kiểm chứng qua **11+ hypotheses**, loại bỏ từng cái, và xác nhận bằng test trực tiếp [DBG-12]:

| Test | Case 1: F5 → FREEZE | Case 2: No F5 → OK |
|------|---------------------|---------------------|
| `AnimateRunner.complete()` gọi | 8 lần | 176 lần |
| `_tick()` gọi | 8 lần | 176 lần |
| `_tickCallback` (callback THỰC SỰ fire) | **0** ❌ | **176** ✅ |
| `_resolve()` gọi | **0** ❌ | **176** ✅ |
| **[DBG-12] test trực tiếp waitQueue** | **🔴 STUCK** | 🟢 OK |
| `onComplete` fired? | **❌ KHÔNG** | **✅ CÓ** |

**[DBG-12] test**: Tạo 1 test callback qua `$$animateAsyncRun()`, push vào waitQueue, chờ 200ms xem có fire không → kết quả **"🔴 waitQueue STUCK!"** → **Root cause confirmed 100%**.

**Các giả thuyết đã loại bỏ:**

1. ❌ `$animate.enabled()` bị disabled
2. ❌ Template popup quá phức tạp (thử template minimal → vẫn freeze)
3. ❌ Tab bị hidden (rAF blocked)
4. ❌ `requestAnimationFrame` bị blocked
5. ❌ `$timeout` bị blocked
6. ❌ Thiếu `$digest` cycle
7. ❌ `.ng-animate` elements blocking `$$rAFScheduler`
8. ❌ Dialog khác đang pending
9. ❌ WebSocket `$apply()` flood
10. ❌ `$$isDocumentHidden()` cache stale
11. ❌ Controller `init()` code gây lỗi

## 2. PHÂN TÍCH VẤN ĐỀ — Dialog bị freeze ở 2 chỗ

Khi `waitQueue` bị stuck, dialog bị ảnh hưởng ở **2 mức**:

### 2.1. init() không được gọi → form trống rỗng

Code gốc trong `popupAddEditTaskPtE.js` (dòng 515):
```javascript
// init();   ← BỊ COMMENT OUT (do debug trước đó)
$scope.$initFromDialog = init;  ← Debug code, lẽ ra không nên có
```

`init()` set `$scope.initialized = true` sau khi load data. Template popup (`popupAddEditTaskPtE.html`) sử dụng `ng-if="initialized"` trên **nhiều field quan trọng**:
- `ng-if="initialized"` → DateTime fields (dòng 36, 41)
- `ng-if="initialized"` → Patient dropdown (dòng 59)
- `ng-if="initialized"` → AssignedTo dropdown (dòng 121)
- `ng-if="initialized"` → Note/CKEditor (dòng 150)

Khi `init()` không chạy → `initialized = false` → **hầu hết form fields không hiển thị**.

### 2.2. $mdDialog.hide() không hoạt động → không đóng được dialog

`$mdDialog.hide()` nội bộ cũng dùng animation. Khi waitQueue stuck → hide animation cũng stuck → dialog **không đóng được**.

Trong controller, `$mdDialog.hide()` được gọi ở 3 chỗ:
- `save()` sau khi save thành công (dòng 469)
- `updateTaskStatus()` (dòng 481)
- `cancel()` (dòng 499)

→ Tất cả đều fail khi waitQueue stuck.

### 2.3. Trang Tasks & Alerts nặng → tăng rủi ro waitQueue stuck

Template `tasksManagementPtE.html` (332 dòng) có DOM phức tạp:

**Left panel — Table:**
- `ng-repeat="task in tasks"` với 10 columns mỗi row + action buttons
- Header sort, search box, pagination, dropdown-box-pte

**Right panel — Filter (luôn compile dù ẩn bằng CSS):**
- 7x `dropdown-list-multiple-pte` (custom directive nặng)
- 1x `md-contact-chips` (Angular Material directive)
- 2x `time-range-input` (custom directive)
- Tổng: ~10 custom directives compile đồng loạt khi page load

Right panel dùng `ng-class="{'open': flagshowPanel}"` (chỉ ẩn/hiện bằng CSS class) → **LUÔN được compile vào DOM** dù user không mở filter. Khi có double route change (F5 scenario), tất cả các directives này compile 2 lần → nhiều animation callbacks hơn → tăng rủi ro waitQueue throw error.

## 3. WORKAROUND — Solution A: Bypass Animation Dependency

### Nguyên tắc

```
┌─────────────────────────────────────────────────────────────────────┐
│  HIỆN TẠI (BỊ LỖI):                                               │
│  $mdDialog.show() → animation → onComplete → init() → render form │
│                          ↑ STUCK                                    │
│                                                                     │
│  SAU FIX (SOLUTION A):                                              │ 
│  $mdDialog.show() → controller instantiate → init() → render form  │
│                      (không phụ thuộc animation)                    │
│                                                                     │
│  + Fallback force-close khi $mdDialog.hide() bị stuck              │
│  + Defer heavy DOM trên trang Tasks để giảm rủi ro                 │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.1. Thay đổi 1 — Restore init() trực tiếp trong Controller

**File:** `app/scripts/controllers/tasks/popupAddEditTaskPtE.js`  
**Dòng:** 515–518

**Trước (code hiện tại — BỊ LỖI):**
```javascript
        // init();
        // init() sẽ được gọi từ onComplete của $mdDialog
        // để đảm bảo animation lock đã release trước khi load data
        $scope.$initFromDialog = init;
```

**Sau (fix):**
```javascript
        init();
```

**Giải thích:**
- `init()` là code gốc — được gọi trực tiếp khi controller instantiate.
- `$scope.$initFromDialog = init` là debug code đã thêm trước đó, cần xóa.
- `init()` gọi `loadTaskGeneralData().then()` → data load xong → set `$scope.initialized = true` → các field có `ng-if="initialized"` hiển thị.
- Controller được instantiate ngay khi `$mdDialog.show()` compile template — **TRƯỚC** animation → **không phụ thuộc waitQueue**.

### 3.2. Thay đổi 2 — Force-close dialog khi $mdDialog.hide() bị stuck

**File:** `app/scripts/controllers/tasks/popupAddEditTaskPtE.js`  
**Dòng:** 497–513 (hàm `cancel()`) + thêm helper function

**Trước (code hiện tại):**
```javascript
        $scope.cancel = function () {
            console.log('cancel-result');
            $mdDialog.hide()
            //-----
            // var result = $mdDialog.hide()
            // ... (debug code commented out) ...
        }
```

**Sau (fix):**
```javascript
        // Helper: force-close dialog khi animation stuck
        function forceCloseDialog() {
            if ($element && $element[0]) {
                var container = $element[0].closest('.md-dialog-container');
                if (container) {
                    angular.element(container).remove();
                }
            }
            // Nếu không còn dialog nào, cleanup backdrop & body class
            if (!document.querySelectorAll('.md-dialog-container').length) {
                angular.forEach(document.querySelectorAll('md-backdrop'), function(el) {
                    angular.element(el).remove();
                });
                document.body.classList.remove('md-dialog-is-showing');
            }
        }

        $scope.cancel = function () {
            var hideResult = $mdDialog.hide();
            // Fallback: nếu hide không close được trong 500ms (animation stuck), force-close
            var fallbackTimer = setTimeout(forceCloseDialog, 500);
            if (hideResult && hideResult.then) {
                hideResult.then(function() { clearTimeout(fallbackTimer); })
                          .catch(function() { clearTimeout(fallbackTimer); });
            }
        }
```

**Cập nhật `save()` và `updateTaskStatus()` tương tự:**

```javascript
        // Trong save() — dòng 469:
        // Trước:  $mdDialog.hide(response);
        // Sau:
        var hideResult = $mdDialog.hide(response);
        var fallbackTimer = setTimeout(forceCloseDialog, 500);
        if (hideResult && hideResult.then) {
            hideResult.then(function() { clearTimeout(fallbackTimer); })
                      .catch(function() { clearTimeout(fallbackTimer); });
        }

        // Tương tự cho updateTaskStatus() — dòng 481
```

**Giải thích:**
- `$mdDialog.hide()` nội bộ trigger hide animation → cũng dùng waitQueue → cũng bị stuck.
- Fallback 500ms: đủ thời gian cho animation bình thường (CSS transition 0.4s) nhưng không quá lâu gây UX lag.
- `forceCloseDialog()` chỉ xóa dialog container CỤ THỂ (qua `$element.closest()`), không ảnh hưởng dialog khác (vì config dùng `multiple: true`).
- cleanup backdrop & body class chỉ khi dialog cuối cùng bị đóng.

### 3.3. Thay đổi 3 — Cleanup dialog config trong app.js

**File:** `app/scripts/app.js`  
**Hàm:** `showPopupAddEditTask()` (~dòng 4995)

**Sau (fix) — toàn bộ hàm:**

```javascript
        function showPopupAddEditTask(task, mode, callback, patientDataView) {

                callback && callback(result);
            }).catch(function (error) {
                console.log('showPopupAddEditTask error', error);
            });
        }
```

**Giải thích:**
- Xóa toàn bộ debug logs ([DBG], [DBG-7], [DBG-8], [DBG-9], [DBG-10], [DBG-12]).
- Xóa `onShowing` — không cần vì `init()` đã chạy trực tiếp trong controller.
- Xóa `onComplete` — không cần vì `init()` không còn phụ thuộc vào nó.
- Config giữ nguyên: `multiple: true`, `skipHide: true`, `focusOnOpen: false`.

### 3.4. Thay đổi 4 — Defer heavy DOM trên trang Tasks & Alerts

**Mục đích:** Giảm số lượng custom directive compile đồng loạt khi page load → giảm animation callbacks → giảm rủi ro waitQueue throw error khi double route change.

#### 3.4.1. Template — tasksManagementPtE.html

**Thay đổi:** Wrap nội dung right panel trong `ng-if="filterPanelReady"`.

**Trước (dòng 148–150):**
```html
    <div class="right-filter-panel tasks-right-filter-panel" ng-class="{'open': flagshowPanel}">
        <div class="position-relative w-100 h-100 d-flex flex-column">
            <div class="d-none d-xl-flex align-items-center ...
```

**Sau:**
```html
    <div class="right-filter-panel tasks-right-filter-panel" ng-class="{'open': flagshowPanel}">
        <div class="position-relative w-100 h-100 d-flex flex-column" ng-if="filterPanelReady">
            <div class="d-none d-xl-flex align-items-center ...
```

**Chỉ thêm `ng-if="filterPanelReady"` vào thẻ div bên trong.**

#### 3.4.2. Controller — tasksManagementPtE.js

**Thay đổi:** Set `filterPanelReady = true` via `$timeout` sau khi page render xong.

**Trong hàm `initializer()` (dòng 119):**

**Trước:**
```javascript
        function initializer() {
            $scope.initializer = true;
            if ($rootScope.screenType === ScreenTypeConst.Phone && $scope.flagshowPanel) {
                $rootScope.hideHeaderLogin = true;
                $rootScope.hideNav = true;
                $('.top-ul').hide();
            }
            Promise.all([loadTaskUserPreferences(), loadTaskGeneralData()]).then(() => initFilterDefault());
        }
```

**Sau:**
```javascript
        function initializer() {
            $scope.initializer = true;
            $scope.filterPanelReady = false;
            if ($rootScope.screenType === ScreenTypeConst.Phone && $scope.flagshowPanel) {
                $rootScope.hideHeaderLogin = true;
                $rootScope.hideNav = true;
                $('.top-ul').hide();
            }
            Promise.all([loadTaskUserPreferences(), loadTaskGeneralData()]).then(() => {
                initFilterDefault();
                // Defer right panel compilation sang tick tiếp theo
                // Giảm animation callbacks trong lần render đầu tiên (route transition)
                $timeout(function() {
                    $scope.filterPanelReady = true;
                }, 0);
            });
        }
```

**Lưu ý:** Cần inject `$timeout` vào controller nếu chưa có.

**Giải thích:**
- Right panel có ~10 custom directives (dropdown-list-multiple-pte, md-contact-chips, time-range-input).
- Hiện tại: tất cả compile đồng loạt khi page load, dù panel ẩn (CSS `ng-class` chỉ ẩn visual, không ngăn compile).
- Sau fix: Right panel chỉ compile **sau khi data đã load** và **sau 1 tick** (`$timeout(fn, 0)`).
- Khi double route change xảy ra (F5 scenario), lần render đầu tiên sẽ nhẹ hơn nhiều → ít animation callbacks → giảm rủi ro waitQueue error.
- `$timeout(fn, 0)` defer sang macro-task tiếp theo → không chặn initial render → user không nhận thấy delay (< 16ms).

## 4. ĐÁNH GIÁ TRADE-OFF

### 4.1. UX Impact

| Thay đổi | UX Impact | Giải thích |
|----------|-----------|------------|
| Restore `init()` | ✅ Không thay đổi | Code gốc là `init()`. Chỉ phục hồi lại. |
| Force-close fallback | ✅ Gần như không thay đổi | Trường hợp bình thường: `$mdDialog.hide()` hoạt động, fallback bị cancel. Trường hợp stuck: dialog đóng sau 500ms thay vì không đóng được. User cảm nhận delay ~100ms so với bình thường (CSS transition 400ms + 500ms timeout = 900ms max, thay vì 400ms bình thường). |
| Xóa `onShowing`/`onComplete` | ✅ Không thay đổi | Debug code, không ảnh hưởng UX. |
| Defer right panel | ✅ Không nhận thấy | Right panel compile sau data load (vốn đã async). Chỉ thêm 1 tick delay (~0-16ms). Panel đóng by default nên user không thấy. |

**Tổng kết UX:** Không có thay đổi UX nào mà user nhận thấy.

### 4.3. Risk Assessment

| Risk | Mức độ | Mitigation |
|------|--------|------------|
| `init()` chạy trước animation complete → form field interact trước khi dialog fully visible | Thấp | Dialog có CSS `md-dialog { opacity: 0 }` và `md-transition-in { opacity: 1 }` → dialog visible ngay khi `md-transition-in` class được add (trước animation complete). Form fields hầu hết chờ `initialized = true` (async data load) nên render sau dialog visible. |
| `forceCloseDialog()` gây memory leak / scope leak | Thấp | `angular.element(container).remove()` sẽ trigger Angular's cleanup (destroy scope, unbind watchers). Đây là cách AngularJS xử lý DOM removal. |
| `forceCloseDialog()` ảnh hưởng dialog khác (do `multiple: true`) | Thấp | Dùng `$element[0].closest('.md-dialog-container')` để tìm container CỤ THỂ, không ảnh hưởng dialog khác. |
| Defer right panel gây flash khi user mở filter nhanh | Rất thấp | `$timeout(fn, 0)` chạy sau data load (vốn đã ~200-500ms). Panel default đóng. User phải chủ động click mở filter. |
| Save/UpdateTaskStatus callback trong app.js không được gọi khi force-close | Trung bình | Khi `forceCloseDialog()` xóa DOM, promise từ `$mdDialog.show()` sẽ reject → rơi vào `.catch()`. Cần đảm bảo `callback` trong `.then()` vẫn xử lý đúng (reload task list). Có thể cần thêm `$rootScope.$broadcast('refreshTasksAndAlerts')` trong forceClose. |
