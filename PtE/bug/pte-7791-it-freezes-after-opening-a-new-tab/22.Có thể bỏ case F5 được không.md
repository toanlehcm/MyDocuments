## Câu 1: F5 vs không F5 — khác nhau ở đâu?

**Không F5** (logout → login):
```
Tab 1: Angular SPA vẫn đang chạy → logout/login CHỈ là route change
       → Tất cả JS, directives, services ĐÃ LOAD SẴN
       → Login xong → app settle NHANH (< 500ms)
       → User mở tab 2 → Tab 1 ĐÃ SETTLED → Tab 2 bootstrap đơn độc → OK
```

**Có F5** (logout → F5 → login):
```
Tab 1: F5 = DESTROY toàn bộ page → reload ALL JS/CSS → re-parse Angular modules
       → Re-bootstrap: app.run(), WebSocket, sessionFactory, tất cả từ đầu
       → Login xong → app CHƯA settle (cần 1-3 giây)
       → User mở tab 2 NGAY → CẠNH TRANH tài nguyên CPU với tab 1
       → Tab 2 cũng bootstrap từ đầu → FREEZE
       → Tab 3+ thì OK vì lúc đó tab 1 + tab 2 ĐÃ settle
```

**Root cause của F5 case:** Sau F5, tab 1 cần **GẤP 10 LẦN** thời gian để settle (full re-bootstrap vs just route change). Khi user mở tab 2 quá nhanh, **2 tabs đang bootstrap ĐỒNG THỜI** → browser CPU bị quá tải.

Đây là vấn đề **browser resource contention** — KHÔNG phải lỗi code popup. Muốn fix triệt để F5 case phải làm 1 trong:
- Cross-tab communication (tab 2 chờ tab 1 settle) → phức tạp, dùng localStorage event
- Hoặc giảm toàn bộ bootstrap time của app → refactor lớn
- Hoặc chấp nhận đây là edge case hiếm gặp

## Câu 2: Move init() xuống cuối controller — có ảnh hưởng không?

**KHÔNG ảnh hưởng** gì đến freeze. JavaScript **hoisting** đảm bảo `function init()` luôn available dù gọi ở đâu.

Sự khác biệt duy nhất: init() ở đầu → API call start sớm hơn vài microsecond. Hoàn toàn không đáng kể. Đặt cuối cho dễ đọc code là decision đúng.
