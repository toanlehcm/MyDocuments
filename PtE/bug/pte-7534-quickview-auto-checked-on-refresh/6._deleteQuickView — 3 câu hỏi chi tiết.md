## #1. `_calculateWidthOfQuickView` có cần thêm ở cuối `_deleteQuickView` không?

### Kết luận: ✅ Anh ĐÚNG — KHÔNG cần thêm

### Bằng chứng

`_applyView` trong **cả 2 file** đều đã gọi `_calculateWidthOfQuickView()`:

**invoicePtE.js** (L3916-3918):
```javascript
if ($scope.searchTerms.isEdited) {
  $scope.quickView.selectedQuickView = null;
  _calculateWidthOfQuickView();     // ← đã có
}
```

**insuranceClaimManagementATP.js** (L1723-1725):
```javascript
if ($scope.searchTerms.isEdited) {
  $scope.quickView.selectedQuickView = null;
  _calculateWidthOfQuickView();     // ← đã có
}
```

### ⚠️ Nhưng có 1 điểm cần lưu ý

> [!WARNING]
> `_applyView` **CHỈ** gọi `_calculateWidthOfQuickView()` khi `$scope.searchTerms.isEdited === true`.
>
> Trong flow `_deleteQuickView`:
> - `_deleteQuickView` → `_processSelectQuickView(defaultViewOfSystem)` → `_applyView(false)`
> - Lúc này `isEdited` **KHÔNG được set** (không ai set `isEdited = true` trước `_applyView`)
> - → `_calculateWidthOfQuickView()` **SẼ KHÔNG ĐƯỢC GỌI** qua `_applyView`

### Vậy tag width có bị sai không?

**KHÔNG.** Vì:

1. **Khi xóa active view** (nhánh `view.selected && selectedQuickView === view._id`): Tag bar đã được rebuild từ `$scope.quickView.quickViewList = _.cloneDeep(quickViewList)` (L3474). AngularJS `ng-repeat` sẽ re-render list → browser tự tính lại width → **KHÔNG cần** `_calculateWidthOfQuickView()`.

2. **Khi xóa non-active view**: Tương tự, `quickViewList` đã được rebuild (L3474), `ng-repeat` re-render → OK.

3. **`_calculateWidthOfQuickView` chỉ cần** khi có thay đổi CSS mà AngularJS không detect được (ví dụ: thay đổi `selectedQuickView` highlight → width thay đổi do bold/border). Nhưng trong flow delete, view đã bị **remove khỏi list** chứ không phải thay đổi style.

### So sánh với ATP

ATP `_deleteQuickView` cũng **KHÔNG gọi** `_calculateWidthOfQuickView()` → **Đồng nhất.**

### Verdict

| Tiêu chí | Kết quả |
|----------|---------|
| Đồng nhất với ATP? | ✅ Cả 2 đều không gọi |
| Logic đúng? | ✅ Tag list rebuild qua `ng-repeat` |
| Cần thêm? | ❌ Không |

## #2. `_selectQuickView` làm gì?

### Chức năng chính

`_selectQuickView(view)` ([L3337-3374](file:///d:/Sources/pteverywhere/Client/app/scripts/controllers/insurance/invoicePtE.js#L3337-L3374)) là **hàm xử lý khi user CLICK vào một quick view tag** trên tag bar. Nó là wrapper bao ngoài `_processSelectQuickView` với logic bổ sung.

### Phân rã logic

```
_selectQuickView(view)
│
├─ 1. Guard: isEdited? → reset selectedQuickView + viewId (cho phép re-select)
│
├─ 2. Guard: view null hoặc đã là view hiện tại? → return (không làm gì)
│
├─ 3. Nếu isEdited && có quyền Admin:
│     └─ Hiện bootbox confirm "Bạn có muốn chuyển view?"
│         ├─ Yes → isEdited = false → _processSelectQuickView → _applyView
│         └─ No  → không làm gì
│
└─ 4. Nếu KHÔNG isEdited:
      └─ isEdited = false → _processSelectQuickView → _applyView
```

### So sánh `_selectQuickView` vs `_processSelectQuickView`

| Aspect | `_selectQuickView` | `_processSelectQuickView` |
|--------|-------------------|--------------------------|
| **Mục đích** | Xử lý user interaction (click tag) | Core logic: bind data + render panel |
| **Guard duplicate** | ✅ Check `view._id === viewId` → skip | ❌ Không check |
| **Confirm dialog** | ✅ Hỏi confirm nếu isEdited | ❌ Không |
| **Set isEdited** | ✅ Set `isEdited = false` | ❌ Không set |
| **Set isDefaultOfSystem** | ✅ Set `isDefaultOfSystem` | ❌ Không set |
| **Gọi _applyView** | ✅ Có (fetch data mới) | ❌ Không |
| **Gọi _renderViewToPanelFilter** | Gián tiếp (qua `_processSelectQuickView`) | ✅ Trực tiếp |

> [!NOTE]
> **Tóm lại:** `_selectQuickView` = **user-facing wrapper** (guards + confirm + state management) + `_processSelectQuickView` (core render). `_processSelectQuickView` = **internal helper** chỉ lo render panel, không có guards.

## #3. `_deleteQuickView` gọi `_processSelectQuickView` thay vì `_selectQuickView` — có thiếu logic không?

### Kết luận: ✅ AN TOÀN, KHÔNG thiếu logic — và LÀ ĐÚNG

### Phân tích chi tiết: Tại sao KHÔNG NÊN gọi `_selectQuickView`

#### Lý do 1: Guard duplicate sẽ CHẶN flow

```javascript
// _selectQuickView (L3343)
if (!view || view._id === $scope.searchTerms.viewId) {
  return;  // ← SẼ CHẶN!
}
```

Trong `_deleteQuickView`, trước khi gọi `_processSelectQuickView(defaultViewOfSystem)`, code đã set:
```javascript
$scope.quickView.selectedQuickView = defaultViewOfSystem._id;  // L3482
```

Nếu user **đang ở defaultViewOfSystem** và xóa view khác → `$scope.searchTerms.viewId` có thể đã = `defaultViewOfSystem._id` → **`_selectQuickView` sẽ return sớm**, KHÔNG render gì cả.

#### Lý do 2: Confirm dialog KHÔNG CẦN khi system tự chuyển

`_selectQuickView` hỏi: *"Bạn có muốn chuyển view?"* → Điều này chỉ hợp lý khi **user chủ động click**. Khi system tự fallback sau delete, **KHÔNG cần hỏi confirm** — phải chuyển về default ngay lập tức.

#### Lý do 3: `isEdited` flag không relevant

`_deleteQuickView` đang **xóa tag khỏi quick view bar**, không phải edit filter. Trạng thái `isEdited` của filter panel **không liên quan** đến việc remove tag.

#### Lý do 4: ATP cũng dùng `_processSelectQuickView` trực tiếp

```javascript
// insuranceClaimManagementATP.js (L1905)
_processSelectQuickView(defaultViewOfSystem);  // ← gọi trực tiếp
_applyView(false);
_showFilter();
```

**Cùng pattern, cùng lý do.**

### Bảng so sánh: Nếu dùng `_selectQuickView` vs `_processSelectQuickView`

| Scenario | Dùng `_selectQuickView` | Dùng `_processSelectQuickView` |
|----------|------------------------|-------------------------------|
| Xóa active view → fallback default | ⚠️ Có thể bị guard chặn + popup confirm không cần thiết | ✅ Chuyển thẳng, không hỏi |
| Xóa non-active view | N/A (code không vào nhánh này) | N/A |
| `isEdited = true` khi xóa | ❌ Hiện popup "Bạn muốn chuyển view?" — UX tệ | ✅ Chuyển thẳng |
| `isEdited = false` khi xóa | ⚠️ Guard `viewId === defaultViewOfSystem._id` có thể chặn | ✅ OK |
| Set `isDefaultOfSystem` | ✅ Có set | ⚠️ Không set — **nhưng OK vì defaultViewOfSystem.IsDefaultOfSystem = true** |

### ⚠️ Logic duy nhất bị "skip" khi dùng `_processSelectQuickView`

> [!IMPORTANT]
> `_selectQuickView` set `$scope.searchTerms.isDefaultOfSystem = view.IsDefaultOfSystem` (L3360/3369).
>
> `_processSelectQuickView` **KHÔNG set** `isDefaultOfSystem`.
>
> **Nhưng điều này AN TOÀN vì:**
> - View fallback luôn là `defaultViewOfSystem` → `IsDefaultOfSystem = true`
> - `isDefaultOfSystem` dùng để control UI (ẩn/hiện nút Delete View, Save View)
> - Khi fallback về default view, các nút này nên bị ẩn → **đúng hành vi mong muốn** vì default view = system view = không được xóa/sửa
> - **Kiểm chứng:** `isDefaultOfSystem` chỉ ảnh hưởng UI disable, KHÔNG ảnh hưởng data flow

### So sánh pattern trong codebase

| Nơi gọi | Hàm được gọi | Tại sao? |
|---------|--------------|----------|
| **User click tag** | `_selectQuickView` | User action → cần guards + confirm |
| **`_addMoreView` (thêm view mới)** | `_selectQuickView` (L3540) | User action → cần guards + confirm |
| **`_editQuickView` (edit view)** | `_processSelectQuickView` (L3395/3403) | System action → KHÔNG cần confirm |
| **`_deleteQuickView` (xóa tag)** | `_processSelectQuickView` (L3483) | System fallback → KHÔNG cần confirm |
| **`_resetView`** | `_processSelectQuickView` (L3671) | System action → KHÔNG cần confirm |

> [!TIP]
> **Pattern rõ ràng:** User actions → `_selectQuickView` (có guards). System/internal actions → `_processSelectQuickView` (không guards). `_deleteQuickView` fallback về default là **system action** → dùng `_processSelectQuickView` là **ĐÚNG pattern**.
