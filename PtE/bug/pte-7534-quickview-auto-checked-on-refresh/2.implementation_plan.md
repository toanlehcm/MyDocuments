# Fix: `_deleteQuickView` không persist lên server (PTE-7534)

## Background

`_deleteQuickView()` xóa Quick View tag khỏi UI nhưng **không gọi API** để persist thay đổi.
Sau khi refresh, server trả về dữ liệu cũ → Quick View bị restore.

Ngược lại, `_addMoreView()` (uncheck trong dropdown) luôn gọi `_callAPIAddRemoveQuickView("remove", id)` và rebuild cả list từ response của server. Đây là pattern chuẩn cần áp dụng cho `_deleteQuickView`.

---

## PTE Rule Compliance

> Theo `pte-ai-context.md` và `.agent/code-standards/`:
> - **AngularJS 1.x pattern:** KHÔNG gọi API trong loop, không block `$digest` cycle.
> - **skill-ui-freeze-prevention:** Mọi API call phải async. Không gây freeze khi chờ response.
> - **skill-error-handling:** Phải xử lý error case, dùng `bootbox.alert` nếu cần thông báo user.
> - **skill-null-safety:** Guard `if (!res) return;` trước khi xử lý response.
> - **patterns.md:** Logic phải nằm trong Controller function, không inline trên template.
> - **anti-patterns.md:** Không hard delete — pattern chuẩn là gọi API rồi rebuild list từ server response.

## Ba Options — Trade-off Analysis

### Option A — Pessimistic Update (Mirror `_addMoreView` pattern) ✅ Recommended

Gọi API trước → Chờ server trả về updated list → Rebuild cả `quickViewList` và `allViewList` từ response.
Đây là pattern **đang tồn tại** trong codebase (dùng bởi `_addMoreView`).

```
User click "x" → Tag vẫn còn trên UI (trong vài ms) → API call → Server OK
    → Rebuild list từ response → Tag biến mất
```

| | |
|---|---|
| **UX** | 🟡 Delay nhỏ (~200ms) trước khi tag biến mất — chấp nhận được vì thao tác này không thường xuyên |
| **Risk** | 🟢 Low — hoàn toàn đồng nhất với pattern của `_addMoreView` |
| **Effort** | 🟢 ~15 lines thay đổi |
| **Data Safety** | 🟢 Không có state mâu thuẫn — UI luôn reflect server state |
| **Error Handling** | 🟢 Nếu API fail → list không thay đổi, không crash |
| **Code Style** | 🟢 Consistent với codebase hiện tại |

---

### Option B — Optimistic Update (Xóa UI trước, rollback nếu fail)

Xóa tag ngay lập tức khỏi UI → Gọi API song song → Rollback nếu fail.

| | |
|---|---|
| **UX** | 🟢 Instant feedback — tag biến mất ngay |
| **Risk** | 🔴 High — phải backup state trước khi xóa, implement rollback, handle race conditions |
| **Effort** | 🔴 ~40 lines, cần clone state |
| **Data Safety** | 🟡 Risk of UI-server desync nếu rollback không hoàn hảo |
| **Code Style** | 🔴 Không có pattern này trong codebase — inconsistent |

**Kết luận:** Overkill cho use case này. UX gain không đủ bù đắp complexity và risk.

---

### Option C — Chỉ thêm API call, giữ nguyên UI update logic cũ

Giữ nguyên code UI update hiện tại, thêm `_callAPIAddRemoveQuickView("remove", view._id)` để persist.

| | |
|---|---|
| **UX** | 🟡 Tag biến mất ngay nhưng state có thể mâu thuẫn nếu server trả về khác với UI |
| **Risk** | 🔴 Medium — nếu API fail, UI đã xóa nhưng server chưa xóa → không consistent |
| **Effort** | 🟢 ~5 lines |
| **Data Safety** | 🔴 Không rebuild từ server response — UI có thể lệch server |
| **Code Style** | 🔴 Fire-and-forget pattern — vi phạm error-handling rule |

**Kết luận:** Quick fix nhưng tạo ra bug mới. Không recommend.

---

## Recommendation

**Go with Option A.** Consistent với `_addMoreView`, safe, đúng error-handling pattern.

---

## UI/UX Impact

| Scenario | Hiện tại (bug) | Sau fix (Option A) |
|---|---|---|
| Click "x" → tag biến mất | ✅ Ngay lập tức | 🟡 Sau ~200ms (API round-trip) |
| Refresh sau khi xóa | ❌ Tag xuất hiện lại | ✅ Tag không còn |
| API lỗi (mạng yếu) | N/A | ✅ Tag không bị xóa, user thấy không có gì thay đổi — safe |
| Tag là selectedQuickView | Chuyển về default view | ✅ Giữ nguyên hành vi, thực hiện sau khi API OK |

> [!NOTE]
> Delay ~200ms trên thao tác "xóa Quick View tag" là chấp nhận được vì đây là thao tác cấu hình (không phải thao tác lặp đi lặp lại thường xuyên). Behavior giống hệt khi uncheck trong dropdown (Option A mirror `_addMoreView`).
