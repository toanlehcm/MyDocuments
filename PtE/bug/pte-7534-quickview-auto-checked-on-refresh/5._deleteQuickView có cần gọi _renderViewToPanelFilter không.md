## #1. `_renderViewToPanelFilter` làm gì? Khi nào cần dùng?

### Chức năng chính

`_renderViewToPanelFilter(view)` là **hàm render lại toàn bộ UI filter panel** khi user chuyển sang một view khác. Cụ thể nó làm 4 việc:

| Bước | Hành động | Code |
|------|-----------|------|
| 1 | **Xóa sạch** tất cả filter element cũ trong `#filter-view` | `angular.element("#filter-view").empty()` |
| 2 | **Reset & rebuild** `$scope.searchTerms.filters` từ `view.Filters` | Parse value, mark selected, sort |
| 3 | **Compile & inject** DOM elements mới (date picker, dropdown, search box...) | `_addFilterElementToPanel(value)` + `$compile` |
| 4 | **Rebuild** `$scope.searchTerms.columns` từ `view.DisplayColumns` | Map + sort column list |

### Khi nào được gọi (8 call sites trong invoicePtE.js)

| Call site | Mục đích | Có cần không? |
|-----------|----------|---------------|
| **Init load** (L640, L676, L695) | Lần đầu render view khi vào tab | ✅ Bắt buộc |
| **`_processSelectQuickView`** (L3333) | User click chọn quick view tag | ✅ Bắt buộc |
| **`_resetView`** (L3676) | User bấm Reset View | ✅ Bắt buộc |
| **`_deleteView`** (L3885) | Xóa custom view → fallback về default | ✅ Bắt buộc |
| **`_createView`** (L4054) | Tạo view mới, load default filters | ✅ Bắt buộc |
| **`_deleteQuickView`** (L3428 — commented out) | Xóa tag → nếu đang active → fallback | ⚠️ **Đây là câu hỏi** |

> **Tóm lại:** `_renderViewToPanelFilter` cần dùng **mỗi khi cần thay đổi view đang hiển thị trên filter panel** — tức là thay đổi cả filter lẫn column.


## #3. Đánh giá Trade-off & Rủi ro

### ✅ KHÔNG gọi `_renderViewToPanelFilter` trực tiếp — AN TOÀN

| Tiêu chí | Đánh giá | Chi tiết |
|-----------|----------|----------|
| **Data consistency** | ✅ An toàn | `_processSelectQuickView` đã gọi `_renderViewToPanelFilter` bên trong → filter panel luôn được rebuild đúng |
| **Pattern consistency** | ✅ Đúng pattern | ATP.js (reference code) cũng KHÔNG gọi trực tiếp → 2 file consistent |
| **No duplicate render** | ✅ Tối ưu | Nếu gọi thêm `_renderViewToPanelFilter` → render panel 2 lần → lãng phí, có thể gây flicker UI |
| **Null safety** | ✅ OK | `_processSelectQuickView` tự find view từ list → có guard `if (found)` |
| **AngularJS digest cycle** | ✅ OK | `_applyView(false)` trigger `_getInvoices()` → `$scope` auto-digest |

### ⚠️ Rủi ro tiềm ẩn (thấp)

| Rủi ro | Xác suất | Ảnh hưởng | Giải thích |
|--------|----------|-----------|------------|
| `defaultViewOfSystem` bị null | Rất thấp | Filter panel không render | Chỉ xảy ra nếu server trả data sai khi init. Cả 2 file đều có logic init defaultViewOfSystem từ API |
| `_processSelectQuickView` không tìm thấy view trong `quickViewList` | Thấp | Panel không thay đổi | `defaultViewOfSystem` luôn có trong `quickViewList` vì nó là system view |
| Xóa view KHÔNG phải active view | Không rủi ro | Không ảnh hưởng panel | Code chỉ vào nhánh `if (view.selected && selectedQuickView === view._id)` khi xóa active view |

### 🔍 So sánh với `_addMoreView`

| | `_addMoreView` | `_deleteQuickView` |
|--|----------------|---------------------|
| Gọi `_renderViewToPanelFilter`? | ❌ Không trực tiếp | ❌ Không trực tiếp |
| Gọi gián tiếp qua? | `_selectQuickView` → `_processSelectQuickView` → `_renderViewToPanelFilter` | `_processSelectQuickView` → `_renderViewToPanelFilter` |
| Khi nào trigger? | Khi add view mới + nó được selected | Khi xóa view đang active → fallback default |

> [!NOTE]
> `_addMoreView` cũng **KHÔNG gọi trực tiếp** `_renderViewToPanelFilter`. Nó gọi `_selectQuickView(view)` (L3540), bên trong gọi `_processSelectQuickView(view)` (L3362/3371), bên trong mới gọi `_renderViewToPanelFilter(found)` (L3333). **Cùng pattern.**
