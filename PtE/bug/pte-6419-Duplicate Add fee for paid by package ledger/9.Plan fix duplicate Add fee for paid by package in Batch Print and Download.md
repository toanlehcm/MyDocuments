# Fix: Duplicate "Add fee for paid by package" in Batch Print/Download

## Background

`transformInvoiceContentToTemplate()` is a shared function used by both Batch Print and Batch Download flows. It renders the upcharge fee rows for each invoice in the batch PDF. The bug: it shows **all** historical "Add fee" entries without filtering — the same filter (`idxRefundPkg`) that was already applied to the single-invoice preview is completely absent here.

**Buggy lines:** `v2_PatientInvoiceLogic.js` line **7399–7410**

---

## User Review Required

> [!IMPORTANT]
> `transformInvoiceContentToTemplate` is a **shared function** used by multiple flows. Any change here affects:
> - Batch Print (`batchPrintPDFAllInvoicePatientWithTermATP`)
> - Batch Download (`getBatchFilePDFInvoiceInfo`)
> - Possibly other callers (need to verify)

### Option A — Inline fix (Recommended ✅)
Apply the same `idxRefundPkg` + `isPackageFee` logic directly inside `transformInvoiceContentToTemplate` at lines 7399–7410, mirroring what was done in the single-preview function.

**Changes:**
- Replace lines 7403–7409 in `transformInvoiceContentToTemplate`
- Also add null guards: `|| 0` for `NewRemainAmount`/`RemainAmount`, and null check for `TransactionAmount` before regex

```
BEFORE (line 7404):
  if (txn.TransactionType == 'Upcharge'

  })
  ...

      && jindex > idxRefundPkg
  if (txn.TransactionType == 'Upcharge' || isPackageFee)
```

|---|---|

### Option B — Extract shared helper function
Extract the entire upcharge-fee rendering block into a private helper `_buildUpchargeFeeHtml(invoiceData, currencyFormat)`, and call it from both `printPDFInvoicePatientATP` and `transformInvoiceContentToTemplate`.

| | |
|---|---|
| **Risk** | 🟡 Medium — touches 2 functions + adds new function |
| **Effort** | 🟡 Medium — ~30 lines refactored |
| **Maintainability** | 🟢 High — single source of truth, no duplication |
| **Side effects** | Low — pure function, no state dependency |

---

### Option C — Unify rendering paths
Refactor `printPDFInvoicePatientATP` to internally call `transformInvoiceContentToTemplate` instead of having its own rendering copy.

| | |
|---|---|
| **Risk** | 🔴 High — large structural change, high regression surface |
| **Effort** | 🔴 Large — major refactor |
| **Maintainability** | 🟢 Highest — single rendering path |
| **Side effects** | Could affect single-invoice PDF layout if differences exist between the two paths |

---

## Recommendation

**Go with Option A** for this bug fix (immediate, safe, targeted).
**Plan Option B** as a follow-up refactor task to eliminate code duplication long-term.

Option C is too risky without a dedicated regression test suite.

|---|---|---|
| Batch Print — invoice paid by package, then refunded, then re-paid | Shows ALL "Add fee" entries (duplicates visible in PDF) | Shows ONLY the latest "Add fee" entry after the last refund |
| Batch Download | Same duplicate issue | Fixed same way |
| Single Invoice Preview | Already correct (fixed previously) | No change |
| Summarized batch list (`summariezedListOnly = true`) | Not affected — uses a different table rendering path | Not affected |

> [!NOTE]
> The fix is **invisible to users who don't have the refund-then-repay scenario**. For affected invoices, the duplicate fee rows will disappear from the batch PDF, matching what the single-preview already shows.
