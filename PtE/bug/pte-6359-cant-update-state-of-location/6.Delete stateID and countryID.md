## `null` vs `delete` cho ObjectId field

**Về mặt Mongoose**: Gán `null` cho non-required ObjectId field **KHÔNG gây CastError**. Nó hợp lệ — MongoDB sẽ lưu `null` vào document.

**Nhưng có 2 vấn đề thực tế:**

### 1. `null` sẽ **ghi đè** giá trị cũ trong DB
```
// Giả sử DB đang có:
AddressDetails.CountryId = ObjectId("abc123...")

// Nếu gán null → server gửi $set CountryId: null
// DB sau khi save:
AddressDetails.CountryId = null  ← mất giá trị cũ!
```

Với `delete`, field không có trong payload → server không `$set` → **DB giữ nguyên giá trị cũ**.

### 2. Code khác có thể vẫn dùng `.populate('CountryId')`
Nếu bất kỳ chỗ nào trong hệ thống (reports, exports, queries) vẫn dùng `CountryId` để `.populate()`, gán `null` sẽ khiến populate trả về `null` → có thể gây lỗi ở những nơi đó.

## Kết luận

| Approach | Mongoose hợp lệ? | An toàn cho hệ thống? | Đúng ý anh? |
|----------|:-:|:-:|:-:|
| `= null` | ✅ | ⚠️ Ghi đè DB, populate hỏng | ❌ |
| `= ""` | ❌ CastError | ❌ | ❌ |
| `delete` | ✅ | ✅ Giữ nguyên DB | ✅ |

**`delete` vẫn là đúng nhất** vì:
- Anh nói "hệ thống không còn dùng mấy field `...Id` nữa" → **đừng đụng vào nó**, để nó yên trong DB
- Nếu sau này cần dùng lại, data cũ vẫn còn nguyên
- Đúng rule **null-safety**: không gán giá trị không cần thiết vào ObjectId field

Code hiện tại dùng `delete` là **đúng rule** rồi anh. Không cần đổi sang `null`.

```js
        // State & Country are free-text fields — no reverse-lookup needed.
        // Remove ObjectId fields (CountryId/StateId) from payload to avoid
        // Mongoose CastError when text doesn't match any DB record.
        // Using `delete` is safest: null/""/undefined can all cause ObjectId cast issues.
        if (locationData && locationData.AddressDetails) {
            delete locationData.AddressDetails.CountryId
            delete locationData.AddressDetails.StateId
        }
```
