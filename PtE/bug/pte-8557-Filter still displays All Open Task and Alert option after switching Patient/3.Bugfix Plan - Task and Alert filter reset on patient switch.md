# Kế hoạch sửa lỗi - reset bộ lọc Task/Alert khi chuyển bệnh nhân

## 1) Tóm tắt nguyên nhân gốc (theo dòng code)

### A. Mẫu binding UI đang mong manh (nguồn chính gây lệch hiển thị)
- File: `Client/app/views/patientDashboard/profile-partial/open-tasks-and-alert.html`
- Dòng: `27-37`
- Mẫu hiện tại:
  - `md-radio-group ng-model="filterOf"` tại dòng `27`
  - Thay đổi lựa chọn được xử lý bằng thẻ anchor bọc ngoài:
    - dòng `28`: `ng-click="showDropDownAction($event, 'only')"`
    - dòng `32`: `ng-click="showDropDownAction($event, 'all')"`
- Vì sao rủi ro:
  - Trạng thái radio bị điều khiển một phần bởi model `md-radio-group` và một phần bởi click custom trên thẻ `<a>` bên ngoài.
  - Trong vòng đời chuyển bệnh nhân, model có thể được reset về `only` nhưng trạng thái checked hiển thị vẫn giữ giá trị cũ (`all`) trong dropdown.
  - Điều này giải thích hiện tượng: log API dùng `only`, nhưng UI vẫn highlight `All Open Tasks & Alerts`.

### B. Reset default đang gián tiếp, không đảm bảo cho mọi luồng chuyển bệnh nhân
- File: `Client/app/scripts/controllers/patientDashboard/panelTaskAndAlertController.js`
- Dòng:
  - `26`: init default `filterOf = 'only'`
  - `234`: reset về `only` chỉ trong event handler `refreshTasksAndAlerts`
- Nguồn trigger reset:
  - File: `Client/app/scripts/controllers/patientDashboard/profilePtE.js`
  - dòng `389`: `$rootScope.$broadcast('refreshTasksAndAlerts')`
  - dòng `289`: return sớm khi tab đang chọn không phải `PROFILE`
- Vì sao quan trọng:
  - Cơ chế reset hiện tại phụ thuộc vào luồng event liên controller.
  - Ở một số luồng chuyển bệnh nhân, đường này nhạy với timing/gián tiếp, không phải contract trực tiếp kiểu "đổi bệnh nhân => reset filter".

- `panelTaskAndAlertController.js:26` -> init: `'only'`

## 3) Các phương án fix và trade-off

### Phương án A - Sửa tối thiểu ở UI binding
Thay đổi:
- Thay cơ chế chọn qua anchor bằng cơ chế chọn qua radio-group.
- Dùng `ng-change` trên `md-radio-group` như nguồn sự thật duy nhất.
- Giữ nguyên logic reset theo event hiện tại.

Ưu điểm:
- Patch nhỏ nhất, rủi ro regression thấp.
- Sửa được phần lớn case "model/UI mismatch".

Nhược điểm:
- Vẫn phụ thuộc luồng reset event gián tiếp.
- Chưa chặn race async giữa request cũ/mới.

Ảnh hưởng UI/UX:
- Không đổi UX nhìn thấy, ngoài việc trạng thái chọn trở nên nhất quán.

### Phương án B - Khuyến nghị (cân bằng, an toàn cho product)
Thay đổi:
- Áp dụng Phương án A.
- Trong `PanelTaskAndAlertCtrl`, thêm observer trực tiếp khi switch patient (watch patient id ở scope cha):
  - Khi patient id đổi: ép `filterOf = 'only'`, reset paging, reload.
- Giữ `refreshTasksAndAlerts` event như trigger phụ.
- Thêm request guard trong `fetchTasksAndAlerts()`:
  - theo dõi `currentPatientId` + `requestId`; bỏ qua response cũ.

Ưu điểm:
- Sửa cả 2 nhóm vấn đề gốc: lệch UI binding + contract reset + stale response overwrite.
- Hành vi đúng requirement: luôn default `My Open Task & Alert` sau khi chuyển bệnh nhân.

Nhược điểm:
- Thay đổi lớn hơn một chút và phạm vi test rộng hơn.
- Có thể phát sinh thêm API call nếu watcher patient chưa dedupe.

Ảnh hưởng UI/UX:
- Hành vi bộ lọc ổn định, dễ đoán sau khi switch.
- Không thay đổi layout.

### Phương án C - Hardening đầy đủ
Thay đổi:
- Phương án B + refactor dropdown thành component/service tái sử dụng.
- Bổ sung e2e test coverage cho các kịch bản switch.

Ưu điểm:
- Khả năng bảo trì dài hạn tốt.

Nhược điểm:
- Tốn effort và thời gian QA nhất.
- Không lý tưởng cho hotfix gấp.

Ảnh hưởng UI/UX:
- Không thay đổi chức năng nhìn thấy, nhưng chu kỳ release dài hơn.

## 4) Giải pháp khuyến nghị và lý do

Khuyến nghị **Phương án B**.

Lý do:
- Giải quyết đúng cơ chế lỗi đang xảy ra ở production:
  - trôi trạng thái hiển thị trong radio dropdown
  - reset default chưa gắn trực tiếp với patient change
  - khả năng response async cũ đè response mới khi switch
- Vẫn giữ nguyên UX và phong cách kiến trúc hiện tại (AngularJS 1.x + cấu trúc controller sẵn có) với mức xâm lấn tối thiểu.
