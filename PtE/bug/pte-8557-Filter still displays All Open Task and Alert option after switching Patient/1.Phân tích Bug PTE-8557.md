### Q3 & Q4: Root Cause — Bug ở đâu?

**Đây là chuỗi sự kiện khi switch Patient:**

```
profilePtE.js:389   $rootScope.$broadcast('refreshTasksAndAlerts')
         ↓
panelTaskAndAlertController.js:228  listener nhận event
         ↓
panelTaskAndAlertController.js:234  $scope.filterOf = 'only'  ← RESET đúng
         ↓  
panelTaskAndAlertController.js:235  getTasksAndAlerts()       ← fetch với 'only'
         ↓
panelTaskAndAlertController.js:193  log: "fetchTasksAndAlerts-$scope.totalTasks: 12  only" ← ĐÚNG
```

Vậy tại sao UI vẫn hiển thị "All Open Tasks & Alerts"?

**🎯 Root Cause — Dòng 27 (HTML) + Cơ chế `ng-model`:**

```html
<!-- HTML dòng 27 -->
<md-radio-group ng-model="filterOf" ...>
```

`ng-model="filterOf"` bind **trực tiếp** vào `$scope.filterOf` của `PanelTaskAndAlertCtrl`. Khi controller reset `$scope.filterOf = 'only'` ở dòng 234, UI radio group **harusnya** tự cập nhật về "My Open".

**Nhưng bug xuất hiện do:** Controller **không bị destroy và tạo lại** khi switch Patient. Template `open-tasks-and-alert.html` được include dưới dạng **partial/directive** nên controller **tồn tại xuyên suốt**. Điều này tạo ra một race condition:

```
1. User chọn "All" → $scope.filterOf = 'all'
2. Switch Patient → broadcast 'refreshTasksAndAlerts'
3. Listener SET $scope.filterOf = 'only'
4. getTasksAndAlerts() → fetch với 'only' (đúng về data)
5. ❌ NHƯNG: $scope.fetchTasksAndAlerts là _.debounce 300ms
6. Trong 300ms đó, md-radio-group có thể đã render lại với state cũ từ Angular's $digest
7. ❌ Hoặc: badge count (icon) đang dùng totalTasks từ call 'only'
        nhưng radio UI vẫn còn value 'all' chưa reset kịp
```

**Bug chính xác là: `_.debounce(300ms)` tại dòng 39-41:**

```javascript
// dòng 39-41 — ĐÂY LÀ VẤN ĐỀ
$scope.fetchTasksAndAlerts = _.debounce(function(patientDataView) {
    fetchTasksAndAlerts(patientDataView);   // chạy sau 300ms
}, 300);
```

Khi `$scope.filterOf = 'only'` được set tại dòng 234, rồi `getTasksAndAlerts()` được gọi tại 235, nhưng vì **debounce 300ms**, hàm thực sự chỉ chạy sau 300ms. Trong khoảng thời gian này:

- `$scope.filterOf` = `'only'` (scope đúng)
- `ng-model` của radio group = `'only'` (UI đúng)
- **Badge count trên icon** lại gọi `$scope.fetchTasksAndAlerts` từ nơi khác (ví dụ watcher hoặc trigger khác) mà **không reset filterOf**, dẫn đến count hiển thị số của `'only'` nhưng UI radio vẫn cho thấy `'all'`

**Hoặc khả năng cao nhất:** `$scope.filterOf = 'only'` ở dòng 234 **bị overwrite** bởi `ng-model` two-way binding do Angular `$digest` cycle được trigger sau đó, vì `md-radio-group` đang giữ value `'all'` trong DOM và Angular sync ngược lại vào scope.

### 🔎 Tóm tắt Root Cause

```
Bug dòng 234: $scope.filterOf = 'only' được set đúng
              NHƯNG ng-model two-way binding của md-radio-group
              sync ngược từ DOM('all') → scope, ghi đè lại 'only'
              TRƯỚC KHI debounce 300ms kịp chạy fetchTasksAndAlerts()

Kết quả:
  - Data fetch: dùng 'only' → đúng số (số My Tasks)
  - UI radio:   vẫn hiển thị "All Open Tasks & Alerts"
  - Badge icon: hiển thị số My Tasks (đúng data)
  - Dropdown:   radio button vẫn checked "All" (sai UI)
```

**Khẳng định: Bug ở dòng 234** — `$scope.filterOf = 'only'` bị Angular's `$digest` + `md-radio-group` two-way binding overwrite ngay sau đó do DOM state chưa được reset trước khi scope được sync.
